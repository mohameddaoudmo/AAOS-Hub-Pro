01
المفاهيم الأساسية // CI/CD Fundamentals

قبل ما نبدأ في الـ implementation، لازم نفهم الـ pipeline بشكل كامل — ما هي CI/CD، وليه هي ضرورية لأي مشروع Android جاد.

🔄 Continuous Integration (CI)
  • كل مرة تعمل push للكود، يتعمل build وtest تلقائياً
  • اكتشاف الـ bugs بدري قبل ما تكبر وتتعقد
  • الكود يبقى دايماً في حالة قابلة للـ build
  • فريق العمل متزامن ومافيش merge conflicts كبيرة
🚀 Continuous Delivery (CD)
  • نشر التحديثات بشكل تلقائي للـ testers والمستخدمين
  • إلغاء الخطوات اليدوية الممكن تكون فيها أخطاء
  • الـ release cycle أسرع وأكثر موثوقية
  • Rollback سريع لو حصل مشكلة

💡
ليه Android CI/CD صعبة؟

Android builds بياخدوا وقت طويل (5-15 دقيقة)، الـ emulators محتاجة resources كبيرة، والـ signing process معقدة. CI/CD صح بتحل كل المشاكل دي.

الأداة الدور البديل التقييم
GitHub Actions تشغيل الـ workflow في السحابة GitLab CI، CircleCI، Bitrise مجاني للـ public
Fastlane أتمتة build، test، deploy Gradle scripts مباشرة الأفضل للـ mobile
Firebase Test Lab تشغيل tests على أجهزة حقيقية BrowserStack، Sauce Labs مدفوع بعد الحد
Play Store API نشر تلقائي على Google Play Manual upload يحتاج إعداد
02
GitHub Actions // Workflows, Jobs & Steps

GitHub Actions هي الـ CI/CD platform المدمجة في GitHub. كل حاجة بتتعمل في Workflow files بـ YAML format جوا الـ .github/workflows/ folder.

بنية الـ Workflow

.github/workflows/android.yml 
# ===== اسم الـ workflow يظهر في GitHub UI =====

name: Android CI



# ===== متى يتشغل الـ workflow? =====

on:

  push:

    branches: [ main, develop ]

  pull_request:

    branches: [ main ]

  workflow_dispatch:  # تشغيل يدوي من GitHub UI



# ===== متغيرات البيئة العامة =====

env:

  JAVA_VERSION: 17

  GRADLE_OPTS: "-Dorg.gradle.daemon=false -Dorg.gradle.parallel=true"



# ===== الـ Jobs =====

jobs:

  # ---- Job 1: Lint ----

  lint:

    runs-on: ubuntu-latest

    steps:

      - uses: actions/checkout@v4

      

      - name: Setup JDK

        uses: actions/setup-java@v4

        with:

          java-version: ${{ env.JAVA_VERSION }}

          distribution: temurin

      

      - name: Run Lint

        run: ./gradlew lintDebug

      

      - name: Upload Lint Report

        uses: actions/upload-artifact@v4

        if: always()

        with:

          name: lint-report

          path: app/build/reports/lint-results*.html

Job متكامل للـ Build والـ Test

jobs: build (تكملة الـ workflow) 
  # ---- Job 2: Build ----

  build:

    needs: lint  # يشتغل بعد الـ lint

    runs-on: ubuntu-latest

    

    steps:

      - uses: actions/checkout@v4

      

      # Cache Gradle لتسريع الـ build

      - name: Cache Gradle

        uses: actions/cache@v4

        with:

          path: |

            ~/.gradle/caches

            ~/.gradle/wrapper

          key: ${{ runner.os }}-gradle-${{ hashFiles('**/*.gradle*') }}

          restore-keys: ${{ runner.os }}-gradle-

      

      - name: Setup JDK 17

        uses: actions/setup-java@v4

        with:

          java-version: '17'

          distribution: temurin

          cache: gradle

      

      - name: Grant execute permission

        run: chmod +x gradlew

      

      - name: Run Unit Tests

        run: ./gradlew testDebugUnitTest --stacktrace

      

      - name: Build Debug APK

        run: ./gradlew assembleDebug

      

      # رفع الـ APK كـ artifact

      - name: Upload Debug APK

        uses: actions/upload-artifact@v4

        with:

          name: debug-apk-${{ github.sha }}

          path: app/build/outputs/apk/debug/*.apk

          retention-days: 7

      

      # تقرير الـ Unit Tests

      - name: Upload Test Results

        uses: actions/upload-artifact@v4

        if: always()

        with:

          name: unit-test-results

          path: app/build/reports/tests/

إزاي تتعامل مع الـ Secrets

بيانات حساسة زي كلمات المرور ومفاتيح الـ API لازم تتحط في GitHub Secrets مش في الكود مباشرة.

🚨
قاعدة ذهبية

ما تحطش أي passwords أو API keys أو keystore passwords في الـ YAML file أو الكود. دايماً استخدم GitHub Secrets.

استخدام الـ Secrets في الـ Workflow 
jobs:

  release:

    runs-on: ubuntu-latest

    env:

      # قراءة الـ secrets كـ environment variables

      KEYSTORE_PASSWORD: ${{ secrets.KEYSTORE_PASSWORD }}

      KEY_ALIAS: ${{ secrets.KEY_ALIAS }}

      KEY_PASSWORD: ${{ secrets.KEY_PASSWORD }}

    

    steps:

      - name: Decode Keystore

        run: |

          echo "${{ secrets.KEYSTORE_BASE64 }}" | base64 -d > keystore.jks

          echo "KEYSTORE_PATH=$(pwd)/keystore.jks" >> $GITHUB_ENV

Matrix Strategy — اختبار على نسخ Java مختلفة

Matrix Build Strategy 
jobs:

  test:

    runs-on: ubuntu-latest

    strategy:

      matrix:

        java-version: [ 11, 17, 21 ]

        build-type: [ debug, release ]

      fail-fast: false  # استمر حتى لو فشل واحد

    

    steps:

      - name: Build ${{ matrix.build-type }}

        run: ./gradlew assemble${{ matrix.build-type }}
✅
Gradle Cache = توفير 70% من وقت الـ Build

دايماً استخدم actions/cache مع Gradle. الـ first build بتاخد 8 دقايق، بعد الـ cache بتبقى 2 دقيقة!

Fastlane // Automate Build, Test & Deploy

Fastlane هي أداة Ruby مفتوحة المصدر بتخليك تأتمت كل عمليات الـ mobile development. هي الـ "glue" اللي بتربط كل الأدوات ببعض.

تثبيت Fastlane

Terminal — Installation 
# تثبيت Ruby أولاً (لو مش موجود)

brew install ruby  # macOS



# تثبيت Fastlane

gem install fastlane



# أو عن طريق Bundler (الأفضل للـ CI)

gem install bundler

bundle init



# أضف في Gemfile:

gem "fastlane"



# تثبيت الـ dependencies

bundle install



# تهيئة Fastlane في مشروعك

cd android/

bundle exec fastlane init

بنية ملفات Fastlane

Project Structure 
your-android-project/

├── fastlane/

│   ├── Fastfile       # الـ lanes (الوصفات)

│   ├── Appfile        # بيانات التطبيق

│   └── Pluginfile     # الـ plugins

├── Gemfile            # Ruby dependencies

├── Gemfile.lock

└── app/

Appfile — بيانات التطبيق

fastlane/Appfile 
# Package name بتاع تطبيقك

package_name("com.yourcompany.yourapp")



# JSON key للـ Google Play API

json_key_file("path/to/google-play-key.json")

Fastfile — الـ Lanes الكاملة

fastlane/Fastfile 
# Fastfile — الوصفات الكاملة



default_platform(:android)



platform :android do



  # ===== قبل أي lane =====

  before_all do

    ensure_git_status_clean  # تأكد إن الكود نظيف

  end



  # ===== Lane 1: Tests =====

  desc "تشغيل كل الـ tests"

  lane :test do

    gradle(

      task: "test",

      build_type: "Debug"

    )

  end



  # ===== Lane 2: Beta =====

  desc "Build وnشر نسخة Beta على Firebase"

  lane :beta do

    # زيادة version code تلقائياً

    increment_version_code

    

    # Build الـ APK

    gradle(

      task: "assemble",

      build_type: "Release",

      print_command: false,

      properties: {

        "android.injected.signing.store.file" => ENV["KEYSTORE_PATH"],

        "android.injected.signing.store.password" => ENV["KEYSTORE_PASSWORD"],

        "android.injected.signing.key.alias" => ENV["KEY_ALIAS"],

        "android.injected.signing.key.password" => ENV["KEY_PASSWORD"]

      }

    )

    

    # نشر على Firebase App Distribution

    firebase_app_distribution(

      app: ENV["FIREBASE_APP_ID"],

      groups: "internal-testers, qa-team",

      release_notes: changelog_from_git_commits(

        commits_count: 10,

        pretty: "- %s"

      )

    )

  end



  # ===== Lane 3: Production =====

  desc "Build ونشر على Google Play"

  lane :deploy do

    increment_version_code

    

    # Build AAB (الأفضل للـ Play Store)

    gradle(

      task: "bundle",

      build_type: "Release",

      print_command: false

    )

    

    # رفع على Play Store

    upload_to_play_store(

      track: "internal",  # internal → alpha → beta → production

      aab: "app/build/outputs/bundle/release/app-release.aab",

      skip_upload_screenshots: true,

      skip_upload_images: true

    )

    

    # إشعار Slack

    slack(

      message: "🚀 نسخة جديدة اتنشرت على Internal Track!",

      channel: "#releases"

    )

  end



  # ===== Error Handling =====

  error do |lane, exception|

    slack(

      message: "❌ فشل الـ pipeline في #{lane}: #{exception.message}",

      success: false

    )

  end



end

إزاي تشغل Fastlane من GitHub Actions

.github/workflows/deploy.yml 
name: Deploy



on:

  push:

    tags: [ 'v*' ]  # يشتغل عند إنشاء tag زي v1.2.3



jobs:

  deploy:

    runs-on: ubuntu-latest

    

    steps:

      - uses: actions/checkout@v4

      

      - uses: ruby/setup-ruby@v1

        with:

          ruby-version: '3.2'

          bundler-cache: true  # cache Gems تلقائياً

      

      - uses: actions/setup-java@v4

        with:

          java-version: '17'

          distribution: temurin

      

      - name: Run Fastlane Deploy

        run: bundle exec fastlane deploy

        env:

          KEYSTORE_PATH: ${{ secrets.KEYSTORE_PATH }}

          KEYSTORE_PASSWORD: ${{ secrets.KEYSTORE_PASSWORD }}

          KEY_ALIAS: ${{ secrets.KEY_ALIAS }}

          KEY_PASSWORD: ${{ secrets.KEY_PASSWORD }}

          FIREBASE_APP_ID: ${{ secrets.FIREBASE_APP_ID }}

          FIREBASE_TOKEN: ${{ secrets.FIREBASE_TOKEN }}

Emulator Testing // Instrumented & UI Tests on CI

الـ Instrumented Tests محتاجة device حقيقي أو emulator. تشغيلها على CI هو التحدي الأكبر في Android CI/CD لأن الـ emulator بياخد وقت في الـ startup وبيحتاج RAM كبيرة.

⚠️
Hardware Acceleration مطلوبة

الـ Android Emulator بيحتاج KVM على Linux. GitHub Actions runners بيدعموا ده. على macOS بيستخدم Hypervisor.framework. على Windows أبطأ بكثير.

Workflow كامل لتشغيل Emulator Tests

.github/workflows/instrumented-tests.yml 
name: Instrumented Tests



on:

  push:

    branches: [ main ]

  pull_request:

    branches: [ main ]



jobs:

  instrumented-tests:

    runs-on: ubuntu-latest

    

    steps:

      - uses: actions/checkout@v4

      

      - uses: actions/setup-java@v4

        with:

          java-version: '17'

          distribution: temurin

      

      # تفعيل KVM للـ Hardware Acceleration

      - name: Enable KVM

        run: |

          echo 'KERNEL=="kvm", GROUP="kvm", MODE="0666", OPTIONS+="static_node=kvm"' | sudo tee /etc/udev/rules.d/99-kvm4all.rules

          sudo udevadm control --reload-rules

          sudo udevadm trigger --name-match=kvm

      

      # Cache AVD لتسريع الـ startup

      - name: AVD Cache

        uses: actions/cache@v4

        id: avd-cache

        with:

          path: |

            ~/.android/avd/*

            ~/.android/adb*

          key: avd-api34-${{ hashFiles('**/build.gradle') }}

      

      # إنشاء AVD لو مش في الـ cache

      - name: Create AVD

        if: steps.avd-cache.outputs.cache-hit != 'true'

        uses: reactivecircus/android-emulator-runner@v2

        with:

          api-level: 34

          arch: x86_64

          profile: Nexus 6

          force-avd-creation: false

          emulator-options: -no-window -gpu swiftshader_indirect -noaudio -no-boot-anim -camera-back none

          disable-animations: false

          script: echo "AVD created"

      

      # تشغيل الـ Instrumented Tests

      - name: Run Instrumented Tests

        uses: reactivecircus/android-emulator-runner@v2

        with:

          api-level: 34

          arch: x86_64

          profile: Nexus 6

          force-avd-creation: false

          emulator-options: -no-window -gpu swiftshader_indirect -noaudio -no-boot-anim

          disable-animations: true

          script: ./gradlew connectedDebugAndroidTest --stacktrace

      

      - name: Upload Test Results

        uses: actions/upload-artifact@v4

        if: always()

        with:

          name: instrumented-test-results

          path: app/build/reports/androidTests/

أنواع الـ Android Tests

⚡ Unit Tests — بدون Device
  • بتشتغل على JVM مباشرة — سريعة جداً (ثواني)
  • تغطي الـ business logic والـ ViewModels
  • استخدم Mockito/MockK للـ mocking
  • ./gradlew testDebugUnitTest
🤖 Instrumented Tests — محتاج Device
  • بتشتغل على device/emulator — أبطأ (دقائق)
  • تغطي الـ UI والـ database والـ integrations
  • استخدم Espresso أو UI Automator
  • ./gradlew connectedAndroidTest

مثال Espresso Test احترافي

app/src/androidTest/java/LoginTest.kt 
@RunWith(AndroidJUnit4::class)

@LargeTest

class LoginTest {



    @get:Rule

    val activityRule = ActivityScenarioRule(LoginActivity::class.java)



    @Test

    fun login_withValidCredentials_navigatesToHome() {

        // Given

        onView(withId(R.id.etEmail))

            .perform(typeText("user@test.com"), closeSoftKeyboard())



        onView(withId(R.id.etPassword))

            .perform(typeText("password123"), closeSoftKeyboard())



        // When

        onView(withId(R.id.btnLogin))

            .perform(click())



        // Then

        onView(withId(R.id.homeContainer))

            .check(matches(isDisplayed()))

    }



    @Test

    fun login_withEmptyEmail_showsError() {

        onView(withId(R.id.btnLogin)).perform(click())

        onView(withText("Email مطلوب"))

            .check(matches(isDisplayed()))

    }

}

App Signing & Release Build // Keystore Management

الـ App Signing هي العملية اللي بتثبت إن الـ APK/AAB جه منك أنت. أي تطبيق على Play Store لازم يكون موقّع بـ Keystore ثابت.

🔐
احتفظ بالـ Keystore بأمان!

لو ضيّعت الـ Keystore مش هتقدر تعمل update للتطبيق على الـ Play Store أبداً. اعمل backup في أكتر من مكان آمن.

إنشاء Keystore

Terminal — Create Keystore 
keytool -genkey -v \

  -keystore my-release-key.jks \

  -keyalg RSA \

  -keysize 2048 \

  -validity 10000 \

  -alias my-key-alias



# تحويل الـ keystore لـ Base64 لرفعه في GitHub Secrets

base64 -i my-release-key.jks | tr -d '\n' | pbcopy

# الآن الـ base64 في الـ clipboard، الصقه في GitHub Secret

Signing Config في build.gradle

app/build.gradle.kts 
android {

    signingConfigs {

        create("release") {

            storeFile = file(System.getenv("KEYSTORE_PATH") ?: "debug.jks")

            storePassword = System.getenv("KEYSTORE_PASSWORD") ?: "debug"

            keyAlias = System.getenv("KEY_ALIAS") ?: "debug"

            keyPassword = System.getenv("KEY_PASSWORD") ?: "debug"

        }

    }



    buildTypes {

        release {

            isMinifyEnabled = true

            isShrinkResources = true

            signingConfig = signingConfigs.getByName("release")

            proguardFiles(

                getDefaultProguardFile("proguard-android-optimize.txt"),

                "proguard-rules.pro"

            )

        }

    }

}

Decode وSign في الـ CI

GitHub Actions — Signing Steps 
steps:

  # فك تشفير الـ Keystore من Base64

  - name: Decode Keystore

    run: |

      echo "${{ secrets.KEYSTORE_BASE64 }}" | base64 --decode > ${{ runner.temp }}/keystore.jks

  

  # Build الـ Release APK

  - name: Build Release APK

    run: ./gradlew assembleRelease

    env:

      KEYSTORE_PATH: ${{ runner.temp }}/keystore.jks

      KEYSTORE_PASSWORD: ${{ secrets.KEYSTORE_PASSWORD }}

      KEY_ALIAS: ${{ secrets.KEY_ALIAS }}

      KEY_PASSWORD: ${{ secrets.KEY_PASSWORD }}

  

  # التحقق من الـ Signing

  - name: Verify Signing

    run: |

      apksigner verify --verbose app/build/outputs/apk/release/app-release.apk

Workflow النشر على Firebase

.github/workflows/firebase-distribution.yml 
name: Firebase Distribution



on:

  push:

    branches: [ develop ]  # كل push على develop يوصل للـ testers



jobs:

  distribute:

    runs-on: ubuntu-latest

    

    steps:

      - uses: actions/checkout@v4

        with:

          fetch-depth: 0  # عشان نقدر نعمل git log

      

      - uses: actions/setup-java@v4

        with:

          java-version: '17'

          distribution: temurin

      

      # Build الـ Debug APK

      - name: Build Debug APK

        run: ./gradlew assembleDebug

      

      # عمل Changelog من آخر Commits

      - name: Generate Changelog

        id: changelog

        run: |

          CHANGES=$(git log --oneline -10 --pretty=format:"• %s")

          echo "CHANGELOG=$CHANGES" >> $GITHUB_OUTPUT

      

      # نشر على Firebase

      - name: Upload to Firebase App Distribution

        uses: wzieba/Firebase-Distribution-Github-Action@v1

        with:

          appId: ${{ secrets.FIREBASE_APP_ID }}

          serviceCredentialsFileContent: ${{ secrets.FIREBASE_SERVICE_ACCOUNT }}

          groups: internal-testers

          file: app/build/outputs/apk/debug/app-debug.apk

          releaseNotes: ${{ steps.changelog.outputs.CHANGELOG }}

Best Practices // Performance, Security & Tips

تسريع الـ CI/CD Pipeline

⚡ Gradle Optimization
  • فعّل org.gradle.parallel=true في gradle.properties
  • استخدم org.gradle.caching=true للـ remote cache
  • زيادة الـ heap: org.gradle.jvmargs=-Xmx4g
  • استخدم Configuration Cache (Gradle 8+)
🗄️ Caching Strategy
  • Cache الـ Gradle wrapper وcaches
  • Cache الـ Ruby Gems مع Bundler
  • Cache الـ Android SDK components
  • Cache الـ AVD للـ emulator tests
gradle.properties — إعدادات الأداء 
# تحسينات الأداء

org.gradle.parallel=true

org.gradle.caching=true

org.gradle.configureondemand=true

org.gradle.jvmargs=-Xmx4g -XX:MaxMetaspaceSize=512m -XX:+HeapDumpOnOutOfMemoryError



# Android

android.useAndroidX=true

android.enableJetifier=false

android.nonFinalResIds=true



# Kotlin

kotlin.incremental=true

kotlin.caching.enabled=true

concurrency:

  group: ${{ github.workflow }}-${{ github.ref }}

  cancel-in-progress: true  # إلغاء الـ run القديم عند بدء run جديد



jobs:

  # ======= STAGE 1: Quality Checks =======

  quality:

    runs-on: ubuntu-latest

    steps:

      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4

        with: { java-version: '17', distribution: temurin, cache: gradle }

      - run: ./gradlew lintDebug detekt

      - uses: actions/upload-artifact@v4

        if: always()

        with: { name: quality-reports, path: app/build/reports/ }



  # ======= STAGE 2: Unit Tests =======

  unit-tests:

    runs-on: ubuntu-latest

    steps:

      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4

        with: { java-version: '17', distribution: temurin, cache: gradle }

      - run: ./gradlew testDebugUnitTest koverXmlReport

      - name: Coverage Report

        uses: codecov/codecov-action@v4

        with: { files: app/build/reports/kover/report.xml }



  # ======= STAGE 3: Build =======

  build:

    needs: [ quality, unit-tests ]

    runs-on: ubuntu-latest

    steps:

      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4

        with: { java-version: '17', distribution: temurin, cache: gradle }

      - name: Decode Keystore

        run: echo "${{ secrets.KEYSTORE_BASE64 }}" | base64 -d > ${{ runner.temp }}/key.jks

      - run: ./gradlew assembleRelease bundleRelease

        env:

          KEYSTORE_PATH: ${{ runner.temp }}/key.jks

          KEYSTORE_PASSWORD: ${{ secrets.KEYSTORE_PASSWORD }}

          KEY_ALIAS: ${{ secrets.KEY_ALIAS }}

          KEY_PASSWORD: ${{ secrets.KEY_PASSWORD }}

      - uses: actions/upload-artifact@v4

        with: { name: release-artifacts, path: app/build/outputs/ }



  # ======= STAGE 4: Deploy (فقط على Tags) =======

  deploy:

    needs: build

    runs-on: ubuntu-latest

    if: startsWith(github.ref, 'refs/tags/v')

    environment: production

    steps:

      - uses: actions/checkout@v4

      - uses: actions/download-artifact@v4

        with: { name: release-artifacts }

      - uses: ruby/setup-ruby@v1

        with: { bundler-cache: true }

      - run: bundle exec fastlane deploy

Security Checklist

الإجراء الأمني الأهمية ملاحظة
تشفير الـ Keystore بـ Base64 في Secrets ضروري ما تحطهاش في الـ repository أبداً
استخدام Environment Protection Rules مهم طلب approval للـ production deployments
تفعيل Branch Protection Rules ضروري Required status checks قبل المرج
Dependabot للـ dependency updates موصى به تحديث تلقائي للـ dependencies
SAST scanning مع CodeQL متقدم اكتشاف ثغرات أمنية في الكود
🏆
نصيحة الخبراء — الـ Workflow الأمثل

قسّم الـ pipeline لـ stages واضحة: Quality → Test → Build → Deploy. كل stage تتوقف لو فشل stage قبلها. استخدم concurrency لإلغاء الـ runs القديمة عند بدء run جديد — ده بيوفر وقت وتكلفة كبيرة.
