الأساسيات ما هو VHAL؟
Vehicle Hardware Abstraction Layer — الطبقة اللي بتربط Android بالسيارة
التعريف البسيط

VHAL هو طبقة برمجية في Android Automotive OS (AAOS) — بتعمل كـ "مترجم" بين الـ Android Framework من فوق، والـ Hardware الحقيقي للسيارة من تحت. فكر فيه كـ "بوابة" بين عالم Android وعالم السيارة.

💡
الفكرة الجوهرية

بدل ما كل تطبيق Android يتكلم مع الـ ECUs والحساسات مباشرةً (وهو مستحيل ومهين!)، الـ VHAL بيعمل Abstraction layer — نوحد الواجهة ونخلي Android apps تتكلم بلغة واحدة مع كل أنواع السيارات.

ليه محتاجينه؟

كل شركة سيارات (Toyota, BMW, Ford…) عندها hardware مختلف تماماً. بدون VHAL، كان لازم Android يتغير لكل سيارة. مع VHAL، الـ Android Framework ثابت، والـ VHAL بس هو اللي بيتغير.

وين بيشتغل؟

بيشتغل في الـ IVI (In-Vehicle Infotainment) Head Unit — يعني الشاشة الرئيسية في السيارة. بيتواصل مع الـ Vehicle Network (CAN Bus, LIN, Ethernet) للوصول للـ ECUs والحساسات.

التاريخ والإصدارات

Android 8.0 (Oreo) — 2017

أول إصدار رسمي لـ VHAL كجزء من Android Automotive. الـ interface كانت HIDL-based.

Android 11 — 2020

إضافة properties جديدة للـ EV والـ ADAS. تحسينات في الـ Subscription model.

Android 13 — 2022

التحول من HIDL إلى AIDL (Android Interface Definition Language) — أسرع وأسهل في التعامل.

Android 14+ — 2023 onwards

دعم كامل لـ AIDL VHAL، properties للـ Android Automotive V2U، دعم محسن للـ Cluster Display.

المستقبل

تكامل أعمق مع AI، دعم Over-the-Air updates لـ VHAL properties، تكامل مع SDV (Software-Defined Vehicle).

Properties كل حاجة عن Vehicle Properties
القلب النابض للـ VHAL
الـ Property ده إيه؟

كل حاجة في السيارة ممكن نعبر عنها بـ Property — ليها ID فريد، نوع بيانات، صلاحيات (Read/Write/OnChange)، وقيمة. التفكير في VHAL هو "كل sensor/actuator/state = property واحدة".

تركيب الـ Property ID
Property ID Structure 
// Property ID = 32-bit integer مقسم كالتالي:
// [31:28] = Property Type  [27:16] = Property Group  [15:0] = Property Index

int HVAC_TEMPERATURE_SET = 0x12600504;
// 0x1  = FLOAT type
// 0x26 = HVAC group
// 0x0504 = property index

int VEHICLE_SPEED       = 0x11600207;
int GEAR_SELECTION      = 0x11400400;
int FUEL_LEVEL          = 0x11600307;

Change Mode — متى بيتحدث الـ Property؟
📌
STATIC
بيتحدد مرة واحدة وما بيتغيرش. زي VIN number أو معلومات الهارد وير.
🔔
ON_CHANGE
بيبعت notification بس لما القيمة تتغير. زي Door open/close أو Gear change.
⏱️
CONTINUOUS
بيبعت بـ rate ثابتة (Hz). زي Vehicle speed أو Engine RPM. بيستهلك موارد أكتر.
Access Mode — صلاحيات القراءة والكتابة
READ
قراءة بس — زي Vehicle Speed، Fuel Level، Engine RPM. الـ app مش قادر يغير.
WRITE
كتابة بس — نادر جداً. بيبعت command للـ hardware من غير ما تقرأ الحالة.
READ_WRITE
قراءة وكتابة — زي HVAC Temperature، Seat position. الأكتر شيوعاً.

كل property ليها AreaType بيحدد هي لـ "أنهي منطقة" في السيارة.

قراءة Property (getProperty)
Java / Kotlin 
import android.car.hardware.CarPropertyValue;
import android.car.hardware.property.CarPropertyManager;
import android.car.VehiclePropertyIds;

// 1. الحصول على CarPropertyManager
Car car = Car.createCar(context);
CarPropertyManager carPropMgr =
    (CarPropertyManager) car.getCarManager(Car.PROPERTY_SERVICE);

// 2. قراءة Vehicle Speed
CarPropertyValue<Float> speedProp = carPropMgr.getProperty(
    Float.class,
    VehiclePropertyIds.PERF_VEHICLE_SPEED,  // Property ID
    0                                           // areaId = GLOBAL
);

float speedMs = speedProp.getValue();  // m/s
float speedKmh = speedMs * 3.6f;      // تحويل لـ km/h

// 3. قراءة HVAC Temperature لمقعد السائق
int driverSeat = VehicleAreaSeat.ROW_1_LEFT;
CarPropertyValue<Float> tempProp = carPropMgr.getProperty(
    Float.class,
    VehiclePropertyIds.HVAC_TEMPERATURE_SET,
    driverSeat
);
Log.d("VHAL", "Temp = " + tempProp.getValue());

الاشتراك في التغييرات (registerCallback)
Java / Kotlin — Subscription 
// الـ Callback اللي بيتفعل لما القيمة تتغير
CarPropertyManager.CarPropertyEventCallback speedCallback =
    new CarPropertyManager.CarPropertyEventCallback() {
        @Override
        public void onChangeEvent(CarPropertyValue value) {
            float speed = (Float) value.getValue();
            // بيتفعل كل ما السرعة تتغير
            updateSpeedometer(speed * 3.6f);
        }
        @Override
        public void onErrorEvent(int propId, int areaId) {
            Log.e("VHAL", "Error on property: " + propId);
        }
    };

// اشتراك بـ rate معينة (Hz)
carPropMgr.registerCallback(
    speedCallback,
    VehiclePropertyIds.PERF_VEHICLE_SPEED,
    10.0f  // 10 Hz = 10 updates per second
);

جانب الـ HAL Implementation (OEM Side)
C++ — DefaultVehicleHal.cpp 
// AIDL Interface — ده اللي بيكتبه OEM
class DefaultVehicleHal : public BnVehicle {
public:
    // الـ App طلب قراءة property
    ndk::ScopedAStatus getValues(
        const std::vector<VehiclePropValueRequest>& requests,
        GetValueResults* results) {

        for (auto& req : requests) {
            if (req.prop.prop == toInt(VehicleProperty::PERF_VEHICLE_SPEED)) {
                // إقرأ من CAN Bus أو Sensor
                float speed = readCanBusSpeed();
                results->push_back({.value = speed, .status = StatusCode::OK});
            }
        }
        return ndk::ScopedAStatus::ok();
    }

    // الـ App طلب كتابة property
    ndk::ScopedAStatus setValues(
        const std::vector<VehiclePropValue>& values) {

        for (auto& val : values) {
            if (val.prop == toInt(VehicleProperty::HVAC_TEMPERATURE_SET)) {
                float temp = val.value.floatValues[0];
                // أرسل للـ HVAC ECU عبر CAN Bus
                sendCanMessage(HVAC_TEMP_CMD, temp);
            }
        }
        return ndk::ScopedAStatus::ok();
    }
};

Property Status Codes
Statusالمعنىمتى بيحصل
AVAILABLEالبيانات متاحة وصحيحةالحالة الطبيعية المطلوبة
UNAVAILABLEالبيانات مش متاحة حالياًالـ sensor مش شغال أو معدوم

⚠️
تنبيه مهم جداً

لازم دايماً تتحقق من الـ Status قبل ما تستخدم القيمة. لو UNAVAILABLE وأنت عاملت القيمة كأنها صح، ممكن تحصل مشاكل خطيرة في السيارة!

Status Check Example 
CarPropertyValue<Float> prop = carPropMgr.getProperty(...);

if (prop.getStatus() == CarPropertyValue.STATUS_AVAILABLE) {
    float speed = prop.getValue();
    updateUI(speed);
} else if (prop.getStatus() == CarPropertyValue.STATUS_UNAVAILABLE) {
    showError("Sensor not available");
} else {
    logError("Property error detected");
}

Slow Response
تأخر في تحديث الـ UI → غالباً بسبب Binder thread pool congestion.

Permission Denied

Implementation Blueprints — نماذج التنفيذ
Subscription Multiplexing
تجميع طلبات الـ Subscribe المتشابهة لتقليل الضغط على الـ HAL layer.
Safe Write-Back Pattern
التأكد من نجاح الكتابة عبر مراقبة الـ StatusEvent قبل تحديث الـ UI.
✅ افعل دايماً
✔️ تحقق من getStatus() قبل استخدام أي قيمة
✔️ استخدم registerCallback بدل polling
✔️ Unregister الـ callbacks في onDestroy
✔️ استخدم ViewModel + StateFlow مع الـ callbacks
✔️ تحدد areaId الصح (مقعد/باب/نافذة)
✔️ اختبر على Emulator أولاً بـ inject-vhal-event
🚫 لا تعمل أبداً
✖️ لا تعمل Polling (تكرار getProperty بدون callback)
✖️ لا تستخدم قيمة UNAVAILABLE كأنها صحيحة
✖️ لا تطلب Rate عالية جداً (100Hz+ للـ UI)
✖️ لا تكتب لـ READ-ONLY properties
✖️ لا تعمل blocking call على Main Thread
✖️ لا تتجاهل CarNotConnectedException

💡
قاعدة ذهبية في الـ Rate

الـ UI بيتحدث بـ 60fps (60Hz) — أي rate أعلى من كده هو overhead بلا فايدة. للـ Speedometer مثلاً، 10Hz = كل 100ms = ناعم جداً للعين. لو عملت 100Hz هتاكل battery وCPU بدون أي تحسين مرئي.

الـ OEM Implementation — كيف تعمله؟

كل OEM (Toyota, BMW, Volvo…) بيعمل الـ HAL Implementation الخاص بيه. الخطوات دي بتوضح المسار من الـ Standard VHAL لحد الـ Vehicle Hardware:

OEM HAL Implementation Flow 
// 1. OEM بيـ inherit من IVehicle AIDL interface
class ToyotaVehicleHal : public BnVehicle {
    // 2. بيعمل implement للـ getValues/setValues
    ScopedAStatus getValues(...) override {
        // 3. بيقرأ من الـ CAN Bus أو Vehicle Network
        auto canData = mCanBusClient.read(SPEED_CAN_ID);
        // 4. بيحول الـ raw CAN data لـ VehiclePropValue
        float speedMs = canData.getValue() / 3.6f;
        results->push_back({.value = speedMs});
        return ScopedAStatus::ok();
    }
};

// 5. يسجّل الـ service في init.rc
// service toyota_vhal /vendor/bin/hw/android.hardware.automotive.vehicle
