Vehicle Hardware Abstraction Layer — الطبقة اللي بتربط Android بالسيارة
التعريف التقني

VHAL هو طبقة تجريد برمجية في Android Automotive OS (AAOS) — تعمل كوسيط (Proxy) بين الـ Android Framework من جهة، والشبكة الداخلية للسيارة (Vehicle Network) من جهة أخرى. تهدف لتوحيد الوصول لبيانات السيارة بغض النظر عن نوع الهاردوير.

💡
الفلسفة الهندسية

بدلاً من التعامل المباشر مع بروتوكولات معقدة مثل CAN أو LIN، يوفر الـ VHAL واجهة موحدة تعتمد على مفهوم الـ Properties، مما يسهل تطوير تطبيقات السيارات بشكل مستقل عن الموردين (OEMs).

لماذا التجريد (Abstraction)؟

يسمح الـ VHAL لشركة Google بتطوير نظام AAOS موحد، بينما يقوم كل مصنع سيارات (OEM) بكتابة الـ HAL Implementation الخاص به ليتوافق مع أجهزته، مما يحافظ على استقرار النظام العلوي.

نطاق التشغيل

يعمل الـ VHAL كجزء من الـ Android System، ويتصل عبر طبقة الـ Kernel بمتحكمات الشبكة (Controller Area Network) لاستقاء البيانات من الحساسات أو إصدار أوامر للمشغلات (Actuators).

التطور التاريخي للواجهات

HIDL Era (Android 8 - 12)

استخدام واجهة Hardware Interface Definition Language للربط بين العمليات (IPC).

AIDL Era (Android 13+)

التحول الكامل لـ Android Interface Definition Language لتوحيد المعايير وتحسين الأداء وسهولة التطوير.

البيانات كذرات برمجية
مفهوم الـ Property

كل معلومة في السيارة (سرعة، درجة حرارة، حالة قفل) تُعتبر Vehicle Property. يتم تعريفها بمعرف فريد (ID) يحمل معلومات عن نوع البيانات والمجموعة والمنطقة.

تشريح المعرف (Property ID Structure)
Property ID Mapping 
// المعرف هو 32-bit مقسم لـ 3 أجزاء رئيسية:
// [31:28] = النوع  | [27:16] = المجموعة | [15:0] = الرقم التسلسلي

int PERF_VEHICLE_SPEED = 0x11600207;
// 0x1  -> FLOAT (نوع البيانات)
// 0x16 -> PERF GROUP (مجموعة الأداء)
// 0x0207 -> SPEED INDEX (المعرف الخاص بالسرعة)
STATIC Mode

قيم لا تتغير أبداً مثل سعة خزان الوقود القصوى أو رقم الـ VIN. تُقرأ مرة واحدة فقط.

ON_CHANGE Mode

يتم إرسال إشعار فقط عند تغير القيمة. مثالي لحالات الأبواب، الجير، أو أزرار التحكم.

CONTINUOUS Mode

يتم تحديث البيانات بشكل مستمر بمعدل تردد (Hz) محدد. ضروري للسرعة وضغط المحرك.

نطاق المناطق (Area IDs)

تسمح المناطق بتخصيص الخاصية لجزء معين من السيارة (مثل تكييف المقعد الخلفي يسار).

Area Typeالوصفمثال
GLOBALعلى مستوى السيارة كاملةالسرعة، الوقود
SEATمرتبط بمقعد معينتسخين المقاعد
WINDOWمرتبط بنافذة معينةوضعية النافذة

تفاعل الكود مع النظام
القراءة (Pull Model)
// استخدام CarPropertyManager للحصول على السرعة
CarPropertyValue<Float> speed = carPropMgr.getProperty(
    Float.class,
    VehiclePropertyIds.PERF_VEHICLE_SPEED,
    0 // Global Area
);
float currentSpeed = speed.getValue();
الكتابة (Push Model)
// ضبط درجة الحرارة لمقعد السائق
carPropMgr.setProperty(
    Float.class,
    VehiclePropertyIds.HVAC_TEMPERATURE_SET,
    VehicleAreaSeat.ROW_1_LEFT,
    22.5f
);
الاشتراك (Subscription Model)

أفضل وسيلة للحصول على بيانات محدثة دون استهلاك موارد المعالج في عمليات Polling غير ضرورية.

// التسجيل لاستقبال تحديثات السرعة بمعدل 10Hz
carPropMgr.registerCallback(
    new CarPropertyEventCallback() {
        @Override
        public void onChangeEvent(CarPropertyValue value) {
            // معالجة البيانات الجديدة
        }
    },
    VehiclePropertyIds.PERF_VEHICLE_SPEED,
    10.0f // Hz
);

السيناريو ١
🏎️ تنفيذ عداد السرعة (Real-time Speedometer)
🧭 المبادئ الأساسية للتنفيذ
  • استخدام الـ Property: PERF_VEHICLE_SPEED
  • الترميز البرمجي: الاشتراك (Subscription) هو الخيار الأمثل وليس الـ Polling.
  • ضبط معدل التحديث: معدل 10-20 Hz يوفر سلاسة كافية للعين البشرية دون تحميل النظام.
  • معالجة البيانات: التحقق من الحالة (Status) وتحويل الوحدة من m/s إلى km/h (بالضرب في 3.6).
السيناريو ٢
🌡️ نظام التحكم في المناخ (HVAC Bi-directional Control)
🧭 المبادئ الأساسية للتنفيذ
  • استخدام الـ Property: HVAC_TEMPERATURE_SET
  • تحديد النطاق (Area): استخدام الـ AreaId الصحيح (مثل مقعد السائق: ROW_1_LEFT).
  • التحكم الثنائي: استخدام getProperty للقراءة و setProperty للكتابة.
  • الأذونات: يتطلب إذن android.car.permission.CONTROL_CAR_CLIMATE.
السيناريو ٣
🚪 التحكم المركزي في الأبواب (Central Locking Logic)
🧭 المبادئ الأساسية للتنفيذ
  • استخدام الـ Property: DOOR_LOCK
  • التعامل مع المناطق المتعددة: استخدام getAreaIds() لتحديد كافة الأبواب المدعومة في السيارة.
  • التنفيذ التسلسلي: إرسال أوامر setBooleanProperty لكل باب بشكل منفصل.
السيناريو ٤
🔋 إدارة الطاقة في السيارات الكهربائية (EV Power Management)
🧭 المبادئ الأساسية للتنفيذ
  • حساب النسب المئوية: استخدام EV_BATTERY_LEVEL مقسوماً على EV_BATTERY_CAPACITY.
  • مراقبة الشحن: استخدام EV_CHARGE_PORT_CONNECTED و EV_BATTERY_INSTANTANEOUS_CHARGE_RATE.
  • أفضل ممارسة: عرض البيانات كـ On-change لتجنب استهلاك البطارية عند عدم وجود تغيير.
السيناريو ٥
🔌 إضافة خصائص مخصصة (Custom Vendor Properties)
🧭 خطوات التوسعة والاستمرارية
  • تحديد المعرف: استخدام نطاق الـ Vendor (0x21xxxxxx).
  • التعريف في الـ HAL: إضافة الإعدادات (Config) في كود الـ C++ (AIDL Interface).
  • الربط مع الـ Framework: الوصول للخاصية بنفس الـ ID من خلال الـ Java/Kotlin layer.

إدارة الأذونات (Permissions)

يستخدم AAOS نظام أذونات صارم لحماية بيانات السيارة. لا يمكن للتطبيقات الوصول للـ VHAL دون تعريف الأذونات المناسبة في AndroidManifest.xml.

كفاءة الـ Sampling Rate

طلب ترددات عالية جداً (مثلاً 100Hz+) لبيانات غير حرجة يؤدي لاختناق الـ Binder IPC واستهلاك البطارية. دايماً اختر أقل تردد يحقق تجربة مستخدم جيدة.

أفضل الممارسات البرمجية
  • تحقق من الحالة (Status Check): لا تقم بمعالجة أي قيمة إذا كانت الحالة UNAVAILABLE أو ERROR.
  • إلغاء الاشتراك (Lifecycle): تأكد دايماً من استدعاء unregisterCallback عند إغلاق التطبيق أو الـ Activity لمنع تسريب الذاكرة (Memory Leaks).
  • التعامل مع التأخير (Latency): تذكر أن أوامر الـ VHAL قد تستغرق وقتاً للوصول للـ ECU الفعلي؛ صمم واجهتك لتكون responsive حتى في حالة التأخير.

Pro Tips نصائح هندسية
✅ افعل دايماً
  • ✔️ استخدم الـ Standard Properties كلما أمكن للـ Portability.
  • ✔️ تعامل مع البيانات في Thread خلفي لتجنب تجميد الـ UI.
  • ✔️ اختبر حالات الحواف (Edge Cases) مثل فصل البطارية أو فقدان الإشارة.
🚫 لا تعمل أبداً
  • ✖️ لا تقم بعمل Polling (getProperty) داخل Loops سريعة.
  • ✖️ لا تفترض أن الـ VHAL دايماً متاح (Handling Nulls).
  • ✖️ لا تستخدم الـ VHAL لتخزين بيانات التطبيق الخاصة (للسيارة فقط).
