الأساسيات ما هو VHAL؟
Vehicle Hardware Abstraction Layer — الطبقة اللي بتربط Android بالسيارة
التعريف البسيط

VHAL هو طبقة برمجية في Android Automotive OS (AAOS) — بتعمل كـ "مترجم" بين الـ Android Framework من فوق، والـ Hardware الحقيقي للسيارة من تحت. فكر فيه كـ "بوابة" بين عالم Android وعالم السيارة.

💡
الفكرة الجوهرية

بدل ما كل تطبيق Android يتكلم مع الـ ECUs والحساسات مباشرةً (وهو مستحيل ومهين!)، الـ VHAL بيعمل Abstraction layer — نوحد الواجهة ونخلي Android apps تتكلم بلغة واحدة مع كل أنواع السيارات.

ليه محتاجينه؟

كل شركة سيارات (Toyota, BMW, Ford…) عندها hardware مختلف تماماً. بدون VHAL، كان لازم Android يتغير لكل سيارة. مع VHAL، الـ Android Framework ثابت، والـ VHAL بس هو اللي بيتغير.

وين بيشتغل؟

بيشتغل في الـ IVI (In-Vehicle Infotainment) Head Unit — يعني الشاشة الرئيسية في السيارة. بيتواصل مع الـ Vehicle Network (CAN Bus, LIN, Ethernet) للوصول للـ ECUs والحساسات.

التاريخ والإصدارات

Android 8.0 (Oreo) — 2017

أول إصدار رسمي لـ VHAL كجزء من Android Automotive. الـ interface كانت HIDL-based.

Android 11 — 2020

إضافة properties جديدة للـ EV والـ ADAS. تحسينات في الـ Subscription model.

Android 13 — 2022

التحول من HIDL إلى AIDL (Android Interface Definition Language) — أسرع وأسهل في التعامل.

Android 14+ — 2023 onwards

دعم كامل لـ AIDL VHAL، properties للـ Android Automotive V2U، دعم محسن للـ Cluster Display.

المستقبل

تكامل أعمق مع AI، دعم Over-the-Air updates لـ VHAL properties، تكامل مع SDV (Software-Defined Vehicle).

Properties كل حاجة عن Vehicle Properties
القلب النابض للـ VHAL
الـ Property ده إيه؟

كل حاجة في السيارة ممكن نعبر عنها بـ Property — ليها ID فريد، نوع بيانات، صلاحيات (Read/Write/OnChange)، وقيمة. التفكير في VHAL هو "كل sensor/actuator/state = property واحدة".

Change Mode — متى بيتحدث الـ Property؟
📌
STATIC
بيتحدد مرة واحدة وما بيتغيرش. زي VIN number أو معلومات الهارد وير.
🔔
ON_CHANGE
بيبعت notification بس لما القيمة تتغير. زي Door open/close أو Gear change.
⏱️
CONTINUOUS
بيبعت بـ rate ثابتة (Hz). زي Vehicle speed أو Engine RPM. بيستهلك موارد أكتر.
Access Mode — صلاحيات القراءة والكتابة
READ
قراءة بس — زي Vehicle Speed، Fuel Level، Engine RPM. الـ app مش قادر يغير.
WRITE
كتابة بس — نادر جداً. بيبعت command للـ hardware من غير ما تقرأ الحالة.
READ_WRITE
قراءة وكتابة — زي HVAC Temperature، Seat position. الأكتر شيوعاً.
Area Types — نطاق الـ Property

كل property ليها AreaType بيحدد هي لـ "أنهي منطقة" في السيارة.

// الـ Callback اللي بيتفعل لما القيمة تتغير


CarPropertyManager.CarPropertyEventCallback speedCallback =


    new CarPropertyManager.CarPropertyEventCallback() {


        @Override


        public void onChangeEvent(CarPropertyValue value) {


            float speed = (Float) value.getValue();


            // بيتفعل كل ما السرعة تتغير

// اشتراك بـ rate معينة (Hz)

جانب الـ HAL Implementation (OEM Side)
C++ — DefaultVehicleHal.cpp 
// AIDL Interface — ده اللي بيكتبه OEM


class DefaultVehicleHal : public BnVehicle {


public:


    // الـ App طلب قراءة property


    ndk::ScopedAStatus getValues(


        const std::vector<VehiclePropValueRequest>& requests,


        GetValueResults* results) {





        for (auto& req : requests) {


            if (req.prop.prop == toInt(VehicleProperty::PERF_VEHICLE_SPEED)) {


                // إقرأ من CAN Bus أو Sensor


                float speed = readCanBusSpeed();


                results->push_back({.value = speed, .status = StatusCode::OK});


            }


        }


        return ndk::ScopedAStatus::ok();


    }





    // الـ App طلب كتابة property


    ndk::ScopedAStatus setValues(


        const std::vector<VehiclePropValue>& values) {





        for (auto& val : values) {


            if (val.prop == toInt(VehicleProperty::HVAC_TEMPERATURE_SET)) {


                float temp = val.value.floatValues[0];


                // أرسل للـ HVAC ECU عبر CAN Bus


                sendCanMessage(HVAC_TEMP_CMD, temp);


            }


        }


        return ndk::ScopedAStatus::ok();


    }


};

⚠️
تنبيه مهم جداً

لازم دايماً تتحقق من الـ Status قبل ما تستخدم القيمة. لو UNAVAILABLE وأنت عاملت القيمة كأنها صح، ممكن تحصل مشاكل خطيرة في السيارة!

مسألة نوع ١
🏎️ عايز تعمل Speedometer App يعرض سرعة السيارة في real-time
🧭 طريقة التفكير والحل
  • حدد الـ Property المطلوبة: PERF_VEHICLE_SPEED (FLOAT, READ, CONTINUOUS)
  • لازم تستخدم Subscription (registerCallback) مش polling — عشان real-time
  • حدد الـ rate المناسبة: للـ speedometer عادةً 10-20 Hz كافي
  • تحقق من الـ Status قبل عرض القيمة
  • حوّل من m/s لـ km/h بضرب في 3.6
  • عند onDestroy أو onStop: unregisterCallback عشان ما تتعملش memory leak
Speedometer Implementation 
public class SpeedometerActivity extends CarAppActivity {


    private CarPropertyManager mCarPropMgr;


    private CarPropertyManager.CarPropertyEventCallback mCb;





    @Override


    protected void onCreate(Bundle savedInstanceState) {


        mCb = new CarPropertyManager.CarPropertyEventCallback() {


            @Override


            public void onChangeEvent(CarPropertyValue value) {


                if (value.getStatus() != STATUS_AVAILABLE) return;


                float kmh = (Float) value.getValue() * 3.6f;


                runOnUiThread(() -> mSpeedText.setText(


                    String.format("%.0f km/h", kmh)));


            }


            @Override


            public void onErrorEvent(int propId, int areaId) { }


        };





        mCarPropMgr.registerCallback(


            mCb, VehiclePropertyIds.PERF_VEHICLE_SPEED, 10f);


    }





    @Override


    protected void onDestroy() {


        mCarPropMgr.unregisterCallback(mCb); // مهم جداً!


        super.onDestroy();


    }


}

🌡️ عايز تعمل HVAC Controller — يقرأ الحرارة الحالية ويسمح للمستخدم يغيرها
🧭 طريقة التفكير والحل
  • Property: HVAC_TEMPERATURE_SET — READ_WRITE, ON_CHANGE, FLOAT, per-seat area
  • للقراءة: استخدم getProperty() مع الـ areaId المناسب (مقعد السائق = ROW_1_LEFT)
  • للاشتراك في التغييرات: registerCallback لمعرفة لو حد غيّر من مكان تاني
  • لضبط قيمة جديدة: setProperty() — تأكد إن القيمة في النطاق (عادةً 16-32°C)
  • تعامل مع الـ Permissions: android.car.permission.CONTROL_CAR_CLIMATE
⚠️
علامة مهمة في HVAC

لو شفت الـ property بـ SEAT area وما حددتش الـ areaId الصح، ممكن تغير حرارة مقعد غلط أو تجيلك exception. دايماً حدد السيت المطلوب.

مسألة نوع ٣
🚪 مسألة قفل وفتح الأبواب — عايز تعمل "Lock All Doors" button
🧭 طريقة التفكير والحل
  • Property: DOOR_LOCK — BOOLEAN, READ_WRITE, ON_CHANGE, per-door area
  • الـ areaIds: ROW_1_LEFT, ROW_1_RIGHT, ROW_2_LEFT, ROW_2_RIGHT (وممكن ROW_3_*)
  • احتاج تعرف عدد الأبواب أولاً: استخدم getAreaIds(DOOR_LOCK)
  • Loop على كل areaId واعمل setBooleanProperty(DOOR_LOCK, areaId, true)
  • Permission مطلوب: android.car.permission.CONTROL_CAR_DOORS
مسألة نوع ٤
🔋 مسألة EV — عايز تعرض معلومات البطارية: مستوى الشحن، وقت الشحن المتبقي، هل متوصل بشاحن
🧭 Properties اللي محتاجها
  • EV_BATTERY_LEVEL — FLOAT, مستوى الشحن بالـ Wh
  • EV_BATTERY_CAPACITY — FLOAT, السعة الكلية بالـ Wh (Static)
  • EV_CHARGE_PORT_CONNECTED — BOOLEAN, هل الشاحن متوصل
  • EV_BATTERY_INSTANTANEOUS_CHARGE_RATE — FLOAT, معدل الشحن (mW)
  • النسبة المئوية = (BATTERY_LEVEL / BATTERY_CAPACITY) × 100
  • وقت الشحن = (CAPACITY - LEVEL) / CHARGE_RATE
مسألة نوع ٥
🛠️ مسألة Custom Property — عايز تضيف property جديدة مش موجودة في VHAL الـ Standard
🧭 خطوات إضافة Custom Property
  • حدد الـ Property ID: لازم يبدأ بـ 0x21xxxxxx (Vendor range)
  • عرّف الـ Property في الـ HAL implementation (C++): بحدد النوع، الـ Access Mode، الـ Change Mode
  • أضف الـ Property لـ VehiclePropertyStore
  • في Android side: استخدم CarPropertyManager.getProperty() بنفس الـ ID
  • أضف الـ Permission المناسب في AndroidManifest
  • اعمل VHAL test للـ property الجديدة
🚫
انتبه!

الـ Vendor properties مش portable — لو نقلت الـ App لسيارة تانية بـ OEM تاني، الـ property مش موجودة. دايماً استخدم Standard properties لو ممكن.

→ استخدم STATIC + getProperty() بدون subscription

علامة: Property لمنطقة معينة

لو المسألة بتقول "للسائق" أو "للباب الأمامي" أو "للشباك الخلفي"

→ ده area-based property — لازم تحدد الـ areaId المناسب
🔐
علامة: Access Denied أو SecurityException

الـ app بيرفع exception لما بيحاول يقرأ property

→ Permission مش مضافة في Manifest. أضف الـ car.permission المناسبة.
📡
علامة: القيمة دايماً UNAVAILABLE

الـ property موجودة في الكود بس القيمة دايماً unavailable

→ مشكلة في الـ HAL implementation أو الـ sensor/ECU مش شغال أو الـ areaId غلط
💾
علامة: Memory Leak في الـ App

الـ app بطيء بعد فترة أو بيستهلك RAM كتير

→ ناسي unregisterCallback() في onDestroy/onStop
🎯
علامة: القيمة المتبعتة غلط أو قديمة

الـ callback بيتفعل بس القيمة مش صح

→ تحقق من الـ Timestamp في CarPropertyValue — ممكن القيمة stale. تأكد من getTimestamp()

Pro Tips نصائح من الخبرة
✅ افعل دايماً
  • ✔️ تحقق من Status قبل استخدام أي قيمة
  • ✔️ Unregister الـ callbacks في onDestroy
  • ✔️ استخدم Coroutines أو LiveData مع الـ callbacks
  • ✔️ اختبر على Emulator أولاً بـ inject-vhal-event
  • ✔️ راجع الـ Permissions في Manifest قبل كل test
🚫 لا تعمل أبداً
  • ✖️ لا تستخدم Polling (تكرار getProperty بدون callback)
  • ✖️ لا تستخدم قيمة UNAVAILABLE كأنها صحيحة
  • ✖️ لا تطلب Rate عالية جداً (100Hz+ للـ UI = overhead)
  • ✖️ لا تكتب لـ READ-ONLY properties
  • ✖️ لا تعمل blocking call على Main Thread
