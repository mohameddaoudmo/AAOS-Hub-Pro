01

ما هو CAN Bus؟

💡
CAN = Controller Area Network — بروتوكول اتصال تسلسلي طوّرته شركة Robert Bosch GmbH عام 1983، وأصبح المعيار الدولي ISO 11898. كل السيارات الحديثة تعتمد عليه.
🎯 الغرض الأساسي

قبل CAN، كانت كل ECU متصلة بالأخرى بأسلاك مستقلة (point-to-point). سيارة فيها 70 ECU تحتاج آلاف الأمتار من الأسلاك! CAN حوّل الاتصال إلى شبكة مشتركة بسلكين فقط.

📅 التاريخ والانتشار

ظهر أول تطبيق حقيقي في سيارة Mercedes-Benz W140 عام 1991. اليوم معيار ISO 11898 مطلوب في كل سيارة تُباع في أوروبا والولايات المتحدة منذ 2008.

⚡ المميزات الرئيسية
  • سرعة تصل إلى 1 Mbit/s
  • مقاومة عالية للتداخل الكهرومغناطيسي (EMI)
  • كشف الأخطاء تلقائياً وإعادة الإرسال
  • يدعم حتى 112 عقدة (Node) على نفس الخط
  • لا يوجد Master — كل ECU متساوية
🏗️ أين يُستخدم؟
  • أنظمة المحرك (Engine Control)
  • أنظمة الفرامل ABS / ESP
  • كيس الهواء (Airbag)
  • نظام الهيكل (Body Control Module)
  • الـ Infotainment و ADAS
02

الفيزياء والطبقات — Differential Signaling

قوة CAN تأتي من طريقة إرسال الإشارة الفيزيائية. هو لا يرسل 0 أو 5V بشكل مباشر، بل يستخدم سلكين متعاكسين.

〰️ CAN_H و CAN_L
الحالةCAN_HCAN_Lالفرق (Diff)
Recessive (1)2.5V2.5V≈ 0V
Dominant (0)3.5V1.5V≈ 2V

الـ Dominant يُلغي الـ Recessive — هذا أساس آلية Arbitration.

🔌 Termination Resistors

طرفا الـ Bus لازم يكون فيهم مقاومتين بقيمة 120 Ω لكل منهما. بالتوازي يطلعوا 60 Ω — وده يمنع انعكاس الإشارة (Signal Reflection).

[120Ω] ──── ECU1 ──── ECU2 ──── ECU3 ──── [120Ω]
            CAN_H ━━━━━━━━━━━━━━━━
            CAN_L ─ ─ ─ ─ ─ ─ ─ ─ ─
⚠️
لو قطعت مقاومة الـ Termination أو نسيتها في الـ Prototype، الـ Bus هيبدأ يعمل Noise وهتلاقي Errors كتير جداً — ده من أشهر أخطاء الـ Hardware Bring-up.

CAN نفسه يغطي Layer 1 و 2 بس. البروتوكولات الأعلى (J1939 للشاحنات، CANopen للصناعة، UDS للـ Diagnostics) بتكمّل الباقي.

03

بنية الـ CAN Frame — كل بت وبت

الـ CAN Frame هو الوحدة الأساسية للاتصال. تعالى نشرّح كل جزء فيه.

CAN 2.0A Standard Frame (11-bit ID)

SOF Start 1 bit
Arbitration ID + RTR 11+1 bit
Control IDE+DLC 6 bits
Data Payload 0–64 bits
CRC Checksum 16 bits
ACK ACK 2 bits
EOF End 7 bits
🆔 Message ID (Arbitration Field)

11-bit في Standard (CAN 2.0A) أو 29-bit في Extended (CAN 2.0B). الـ ID مش عنوان المستقبِل — ده معرف المحتوى نفسه. مثلاً:

// أمثلة على IDs شائعة في السيارة 0x0CF → Engine RPM 0x1A0 → Vehicle Speed 0x354 → Steering Angle 0x3B3 → Battery SOC
📦 Data Field (Payload)

من 0 إلى 8 bytes في CAN الكلاسيكي. الـ DLC (Data Length Code) بيحدد الحجم. الشركة المصنّعة بتحدد تفسير كل بايت في ملف اسمه DBC file.

// مثال: رسالة سرعة العجلة ID: 0x1A0 DLC: 8 Byte[0-1]: Speed × 0.01 km/h Byte[2]: Direction flag Byte[3-4]: Odometer Byte[5-7]: Reserved
🔒 CRC — كشف الأخطاء

الـ CRC-15 بيتحسب على كل بتات الـ Frame. كل المستقبِلين بيحسبوه ولو مختلف يرسلوا Error Frame فوراً. CAN قادر يكشف:

  • كل Single Bit Errors
  • كل Double Bit Errors
  • كل Odd Bit Errors
  • Burst Errors لحد 15 bit
✅ ACK — التأكيد

المُرسِل يرسل Recessive bit في حقل الـ ACK. أي مستقبِل استلم الرسالة صح بيحوّله إلى Dominant bit. لو الـ ACK فضل Recessive، المُرسِل يعرف إن محدش استلم وبيعيد الإرسال.

04

أنواع الـ CAN Frames الأربعة

📤 1. Data Frame

الأكثر شيوعاً. ECU بيرسل بيانات حقيقية. RTR bit = Dominant (0).

// ECU_Engine → ALL ID: 0x0CF RTR: 0 DLC: 8 Data: [0x1F, 0xA0, 0x00, ...] // RPM = 0x1FA0 = 8096 RPM
📥 2. Remote Frame (RTR)

ECU بيطلب من ECU تاني يرسلّه داتا معينة. RTR bit = Recessive (1). نادر الاستخدام في السيارات الحديثة.

// ECU_Display يطلب من ECU_Engine الـ RPM ID: 0x0CF RTR: 1 DLC: 8 // No Data — just a request
❌ 3. Error Frame

أي ECU تكشف خطأ ترسل 6 Dominant bits متتالية (Error Flag) لإلغاء الإرسال الحالي، ثم يُعاد الإرسال تلقائياً.

// Error Frame Structure Error Flag: 6 Dominant bits Error Delimiter: 8 Recessive bits // الـ Bus ينتهي من الخطأ خلال ~23µs
⏸️ 4. Overload Frame

ECU محتاجة وقت إضافي لمعالجة رسالة. ترسل Overload Frame لتأخير الإرسال التالي. نادر الاستخدام في التطبيقات الحديثة.

// يُرسَل في Interframe Space فقط Overload Flag: 6 Dominant bits Delimiter: 8 Recessive bits
05

Arbitration — من يتكلم أولاً؟

🧠
الـ Arbitration هو أذكى جزء في CAN. لما ECU-ين يبدأوا يرسلوا في نفس اللحظة، محدش يوقف — بيحصل تنافس تلقائي والأولوية بتتحدد بدون أي تدخل من برنامج.
⚔️ كيف يعمل Bitwise Arbitration
// ECU_A تريد ترسل ID = 0x0CF = 0000 1100 1111 // ECU_B تريد ترسل ID = 0x1A0 = 0001 1010 0000 Bit 10 (MSB): ECU_A: 0 (Dominant) ECU_B: 0 (Dominant) Bus: 0 — TIE — continue Bit 9: ECU_A: 0 (Dominant) ECU_B: 0 (Dominant) Bus: 0 — TIE — continue Bit 8: ECU_A: 0 (Dominant) ← Dominant wins! ECU_B: 1 (Recessive) Bus: 0 — ECU_B يسمع Dominant ≠ ما أرسله → يوقف فوراً! // النتيجة: ECU_A تكمل بدون إعادة إرسال — zero latency! // ECU_B تنتظر وتحاول مرة تانية

⚡ الجميل إن ID أصغر = أولوية أعلى (لأن 0 Dominant يكسب 1 Recessive). يعني ECU_Engine بـ ID=0x000 ليها أعلى أولوية في الـ Network كلها.

📊 Error Counting

كل ECU عندها TEC (Transmit Error Counter) و REC (Receive Error Counter). بيزيدوا عند الأخطاء وبيقلوا عند النجاح.

الحالةTEC/REC
Error Active< 128
Error Passive128 – 255
Bus Off> 255
🚫 Bus Off State

لما TEC يتعدى 255، الـ ECU تفصل نفسها من الـ Bus تماماً (Bus Off). ده حماية لبقية الـ Network. الـ ECU تقدر ترجع بعد 128 occurrences من 11 Recessive bits.

مثال عملي كامل — End-to-End

🎬
سيناريو: السائق يضغط على دواسة البنزين ← RPM يزيد ← التطبيق يعرض RPM الجديد على الشاشة. تعالى نتتبع الداتا خطوة بخطوة.
1️⃣ الخطوة 1: ECM يقيس الـ RPM ويرسل CAN Frame
// ECM (Engine Control Module) بيرسل كل 10ms CAN Frame: ID: 0x0CF // معرف رسالة Engine Data 1 DLC: 8 Data: [0x1F, 0xA0, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00] // حسب DBC file: // Byte 0-1 = RPM × 0.25 → 0x1FA0 = 8096 × 0.25 = 2024 RPM
2️⃣ الخطوة 2: Gateway يفسّر الـ Frame
// Gateway C++ Code CanFrame frame = can_bus.receive(); if (frame.id == 0x0CF) { uint16_t raw_rpm = (frame.data[0] << 8) | frame.data[1]; float rpm = raw_rpm * 0.25f; // = 2024.0 RPM // بعث للـ VHAL daemon vhal_client.setProperty( VehicleProperty::ENGINE_RPM, rpm ); }
3️⃣ الخطوة 3: VHAL Implementation يستقبل ويخزّن
// VHAL C++ Implementation (OEM code) StatusCode MyVhal::set(const VehiclePropValue& propValue) { if (propValue.prop == ENGINE_RPM) { mCurrentValues[ENGINE_RPM] = propValue; // أبلغ كل المشتركين (Subscribers) for (auto& cb : mSubscribers[ENGINE_RPM]) { cb->onPropertyEvent(propValue); } } return StatusCode::OK; }
4️⃣ الخطوة 4: CarService يستقبل ويوزّع على التطبيقات
// CarService (Java/Kotlin) — System Service class CarPropertyServiceClient implements IVehicleCallback { @Override public void onPropertyEvent(VehiclePropValue value) { // يحوّل من VHAL types إلى Android Car API CarPropertyValue<Float> carProp = new CarPropertyValue<>( VehiclePropertyIds.ENGINE_RPM, 0, // area id value.timestamp, (float) value.value.floatValues[0] ); // يبلّغ كل التطبيقات المشتركة notifyListeners(carProp); } }
5️⃣ الخطوة 5: التطبيق يعرض الـ RPM
// Application Code (Kotlin) class DashboardActivity : CarAppActivity() { private val carPropertyManager by lazy { car.getCarManager(Car.PROPERTY_SERVICE) as CarPropertyManager } override fun onCreate(savedInstanceState: Bundle?) { super.onCreate(savedInstanceState) // اشترك في تحديثات الـ RPM carPropertyManager.registerCallback( object : CarPropertyManager.CarPropertyEventCallback { override fun onChangeEvent(value: CarPropertyValue<*>) { val rpm = value.value as Float rpmTextView.text = "${rpm.toInt()} RPM" updateTachometer(rpm) } }, VehiclePropertyIds.ENGINE_RPM, CarPropertyManager.SENSOR_RATE_NORMAL // 1 Hz ) } }
🎉
النتيجة: من لحظة ضغط دواسة البنزين حتى ظهور الـ RPM على الشاشة، في أقل من 10 milliseconds. كل ده عبر CAN Frame → Gateway → VHAL → CarService → App.
📄 بونص: ملف DBC — قاموس الـ CAN

الـ DBC (Database CAN) هو الملف اللي بيعرّف معنى كل بت في كل رسالة. بدونه الداتا مش مفهومة.
