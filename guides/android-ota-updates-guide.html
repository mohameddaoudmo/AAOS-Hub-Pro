01 // INTRO
ما هي OTA Updates؟ // Over-The-Air Update Fundamentals

OTA (Over-The-Air) هي آلية تحديث نظام الأندرويد عبر الإنترنت مباشرةً، بدون ما تحتاج تتصل بكمبيوتر أو تفلاش ملف يدوياً. من أول ما تضغط "Check for Update" لحد ما التحديث يتطبق — في الخلفية منظومة كاملة وعميقة من الكود بتشتغل.

📡
Full OTA
حزمة كاملة تحتوي على كل الـ partitions الجديدة. حجمها كبير (500MB–3GB) لكنها تشتغل من أي إصدار.
📊
Incremental OTA
فقط الـ diff (الفرق) بين الإصدار القديم والجديد. حجمها صغير (20–200MB)، أسرع وأوفر في data.
⚡
Streaming OTA
تطبيق التحديث أثناء التنزيل نفسه (Pixel 3+). لا حاجة لمساحة تخزين إضافية.

تطور آلية التحديث عبر الزمن

ANDROID 1.0 — 2008
Recovery-based OTA
التحديث يتنزل كاملاً، الجهاز يريبوت لـ Recovery، يطبق الحزمة على الـ system partition، ثم يريبوت مرة ثانية. المشكلة: لو فشل = brick!
ANDROID 7.0 (NOUGAT) — 2016
A/B Seamless Updates
ثورة حقيقية. الجهاز عنده نسختين من الـ partitions. التحديث ينزل ويتطبق في الـ background على النسخة غير النشطة. ريبوت واحد سريع فقط.
ANDROID 8.0 (OREO) — 2017
Project Treble
فصل الـ vendor partition عن الـ system. التحديثات الأسرع ممكنة دون انتظار المصنّع.
ANDROID 10 — 2019
Dynamic Partitions
الـ system, vendor, product تصبح partitions ديناميكية قابلة للتغيير الحجم. مرونة أكبر في توزيع المساحة.
ANDROID 11 — 2020
Virtual A/B
A/B بدون مضاعفة المساحة! يستخدم snapshots مؤقتة بدل نسختين كاملتين. أفضل حل حتى الآن.
ANDROID 12+ — 2021
Compression + Updatable APEX
ضغط الـ snapshots لتوفير مساحة + تحديث مكونات الـ system بدون ريبوت عبر APEX modules.
💡
ليه OTA مهمة جداً؟

مليارات الأجهزة Android في العالم. الـ OTA هي الطريقة الوحيدة لإيصال تحديثات الأمان وإصلاح الثغرات على نطاق واسع. تأخير تحديث أمني = ملايين أجهزة معرضة للخطر.

02 // A/B
A/B Partition System // Seamless Updates Architecture

الـ A/B partition system هي أهم إضافة لـ Android internals في السنوات الأخيرة. الفكرة بسيطة عبقرية: الجهاز عنده نسختين من كل الـ critical partitions — نسخة _a ونسخة _b. دايماً واحدة نشطة والثانية "نائمة".

الـ Partitions اللي بتتضاعف في A/B

Partition A/B؟ الوصف الحجم التقريبي
boot نعم Kernel + initramfs. أول ما يشتغل بعد الـ bootloader 32–128 MB
system نعم Android framework, apps النظام, مكتبات Java 1–4 GB
vendor نعم Drivers, HALs, firmware المصنّع 500MB–2GB
product نعم Apps ومحتوى الـ OEM (Google, Samsung, etc.) 100MB–2GB
dtbo نعم Device Tree Blob Overlay للـ hardware ~8 MB
userdata لا بيانات المستخدم — لا تُمس أبداً في OTA باقي الـ storage
misc لا رسائل بين bootloader وOS (update status) ~4 MB
metadata لا معلومات الـ Dynamic Partitions وslot info ~16 MB

كيف تعرف الـ Slot النشط؟

adb shell — Slot Detection Commands

adb shell getprop ro.boot.slot_suffix
# Output: _a  أو  _b

# معلومات مفصلة عن الـ slots
adb shell bootctl get-current-slot
# Output: 0 (يعني _a) أو 1 (يعني _b)

# حالة كل slot

# Slot 0: bootable=yes, successful=yes, tries_remaining=7

# إجبار switch للـ slot الثاني (للتطوير فقط)
adb shell bootctl set-active-boot-slot 1

Virtual A/B — الجيل الثالث (Android 11+)

المشكلة الكبيرة مع A/B الكلاسيكي: مضاعفة المساحة. هاتف 64GB يحتاج 8GB إضافية للـ slot الثاني!

🔁
Traditional A/B Problem
  • نسختان كاملتان من system + vendor + boot
  • تكلفة إضافية 4–8GB مساحة دائمة
  • المساحة محجوزة حتى لو مفيش تحديث
⚡
Virtual A/B Solution
  • slot واحد حقيقي + snapshots مؤقتة في userdata
  • الـ snapshots تُحذف بعد التطبيق الناجح
  • توفير 4–8GB من المساحة المستمرة
Virtual A/B — كيف يعمل dm-snapshot
═══════════════════════════════════════════════════════════
             VIRTUAL A/B — UPDATE FLOW
═══════════════════════════════════════════════════════════

1. قبل التحديث — الـ system partition نشطة عادياً:
   [system partition] ──read──▶ Android OS

2. أثناء التحديث — يتم إنشاء snapshot في userdata:
   [system partition]  ──read──▶ Android OS (يشتغل عادي)
   [system_b snapshot] ◀──write── update_engine (كتابة في الخلفية)
      └─ stored in: /data/gsi/

3. بعد الريبوت — الـ snapshot تصبح النسخة النشطة:
   device mapper يدمج [system partition] + [snapshot]
   
   [snapshot merged] ──read──▶ Android OS الجديد ✓
   
4. بعد التأكيد الناجح — merge ونظّف:
   [system partition] ← merged snapshot data
   [snapshot deleted] — حرير المساحة ✓
03 // ENGINE
Update Engine // update_engine daemon — القلب النابض

update_engine هو الـ daemon المسؤول عن كل شيء: تنزيل الحزمة، التحقق من التوقيع، تطبيق الـ delta، وإدارة الـ slots. كتب بـ C++ ويشتغل كـ system service منذ Android 7.

مكونات Update Engine الداخلية

📡
UpdateAttempter
المنسّق الرئيسي. يدير state machine الكامل من "idle" لـ "updated". يتواصل مع كل المكونات الأخرى.
⬇️
HttpFetcher
تنزيل الحزمة عبر HTTP/HTTPS مع دعم resume، range requests، وتحكم في الـ bandwidth.
🔐
PayloadVerifier
التحقق من توقيع الحزمة بمفتاح RSA مُدمج في الـ system. لا توقيع = لا تطبيق.
⚙️
DeltaPerformer
تطبيق عمليات الـ patch على مستوى الـ blocks. يدعم COPY, ZERO, REPLACE, BSDIFF, PUFFDIFF.
💾
BootControlInterface
التحكم في الـ slots عبر HAL. يحدد الـ active slot، يتحكم في bootable/unbootable flags.
📋
PrefsInterface
تخزين حالة التحديث على الـ disk. لو الجهاز أُغلق أثناء التحديث، يكمل من نفس النقطة.

interface IUpdateEngine {
  // بدء التحديث — يُعطى URL وخصائص الحزمة
  void applyPayload(
    in String url,
    in long   payloadOffset,
    in long   payloadSize,
    in String[] headerKeyValuePairs
  );

  // إلغاء التحديث الجاري
  void cancel();

  // التراجع عن تحديث طبقناه
  void rollback();

  // ربط callback لمتابعة التقدم

  void unbind(in IUpdateEngineCallback callback);

    in ParcelFileDescriptor pfd,
    in long payloadOffset,
    in long payloadSize,
    in String[] headerKeyValuePairs
  );
}

استخدام Update Engine من تطبيق Android (Kotlin)

SystemUpdateManager.kt — تطبيق OTA من الـ App
import android.os.UpdateEngine
import android.os.UpdateEngineCallback

class SystemUpdateManager(
  private val context: Context
) {
  private val updateEngine = UpdateEngine()

  // Callback لمتابعة التحديث
  private val callback = object : UpdateEngineCallback() {

    override fun onStatusUpdate(status: Int, percent: Float) {
      val statusName = when(status) {
        UpdateEngine.UpdateStatusConstants.IDLE                    -> "💤 في الانتظار"
        UpdateEngine.UpdateStatusConstants.CHECKING_FOR_UPDATE    -> "🔍 فحص التحديثات"
        UpdateEngine.UpdateStatusConstants.UPDATE_AVAILABLE       -> "📦 تحديث متاح"
        UpdateEngine.UpdateStatusConstants.DOWNLOADING            -> "⬇️ جاري التنزيل ${(percent*100).toInt()}%"
        UpdateEngine.UpdateStatusConstants.VERIFYING              -> "✅ فحص التوقيع"
        UpdateEngine.UpdateStatusConstants.FINALIZING             -> "⚙️ تطبيق التحديث ${(percent*100).toInt()}%"
        UpdateEngine.UpdateStatusConstants.UPDATED_NEED_REBOOT    -> "🎉 جاهز! أعد التشغيل"
        UpdateEngine.UpdateStatusConstants.ERROR                  -> "❌ خطأ"
        else -> "unknown: $status"
      }
      Log.d("OTA", statusName)
    }

    override fun onPayloadApplicationComplete(errorCode: Int) {
      if (errorCode == UpdateEngine.ErrorCodeConstants.SUCCESS) {
        promptUserToReboot()
      } else {
        handleError(errorCode)
      }
    }
  }

  // تطبيق تحديث من URL
  fun applyUpdate(
    otaUrl: String,
    payloadOffset: Long,
    payloadSize: Long,
    headerProperties: List<String>
  ) {
    updateEngine.bind(callback)
    updateEngine.applyPayload(
      otaUrl,
      payloadOffset,
      payloadSize,
      headerProperties.toTypedArray()
    )
  }
  
  // Rollback لو فيه مشكلة
  fun rollback() {
    updateEngine.rollback()
  }
}

04 // PACKAGES
OTA Package Format // payload.bin, metadata, signatures

كل OTA package هي ملف ZIP يحتوي على بنية محددة جداً. الـ update_engine بيقراها ويطبقها على الـ partitions. خليني أشرح كل عنصر فيها.

بنية الـ OTA ZIP Package

ota_package.zip — Internal Structure
ota_package.zip
├── META-INF/
│   ├── com/android/metadata          # بيانات الحزمة (from/to build)
│   ├── com/android/metadata.pb       # نفس البيانات في Protobuf
│   └── com/android/otacert           # شهادة التوقيع
│
├── payload.bin                        # ⭐ القلب — كل بيانات التحديث
│   ├── DeltaArchiveManifest (protobuf)
│   │   ├── block_size: 4096
│   │   ├── minor_version: 2
│   │   └── partitions: [...]           # قائمة كل الـ partitions
│   │       ├── partition_name: "system"
│   │       ├── new_partition_info: {size, hash}
│   │       └── operations: [...]       # عمليات الـ patch
│   └── partition data (raw blocks)
│
├── payload_properties.txt             # Hashes وحجم الـ payload
│   ├── FILE_HASH=base64(sha256)
│   ├── FILE_SIZE=1234567890
│   ├── METADATA_HASH=base64(sha256)
│   └── METADATA_SIZE=123456
│
└── care_map.pb                        # خريطة الـ blocks المهمة

عمليات الـ Delta (Operations)

Operation Type الوصف متى تُستخدم الكفاءة
SOURCE_COPY نسخ blocks من الـ partition القديم للجديد Blocks لم تتغير ممتاز — بدون بيانات
REPLACE استبدال blocks بمحتوى جديد كاملاً Blocks جديدة كلياً Full size
REPLACE_BZ استبدال بعد ضغط bzip2 بيانات قابلة للضغط متوسط
REPLACE_XZ استبدال بعد ضغط xz (أفضل من bzip2) بيانات متشابهة جيد
PUFFDIFF بيان delta لملفات zlib المضغوطة APKs, JARs بداخل system ممتاز للـ APKs
BSDIFF Binary diff الكلاسيكي Executables, shared libs جيد للـ binaries
ZERO ملء blocks بالأصفار حذف بيانات فوري

05 // SECURITY
Verified Boot & dm-verity // Chain of Trust من الـ Bootloader للـ OS

الـ OTA بدون أمان = كارثة. Verified Boot هي آلية التأكد أن كل مكوّن موقّع وما اتعدّلش. بتشكّل سلسلة ثقة من أول لحظة يشتغل الجهاز.

Chain of Trust — سلسلة الثقة

// VERIFIED BOOT CHAIN
🔒 ROM BOOTLOADER (BL1)
مُثبَّت في الـ hardware. لا يمكن تعديله. يحمل المفتاح العام الجذري.
↓ يُوقِّع ويتحقق
⚡ ABL / BOOTLOADER (BL2)
يتحقق من توقيع الـ boot partition. يتحكم في الـ slot النشط.
↓ يُحمِّل ويتحقق
🐧 KERNEL + INITRAMFS (boot.img)
يُهيئ dm-verity لحماية الـ system partition. يُفعّل AVB metadata.
↓ يُفعّل dm-verity
🤖 SYSTEM + VENDOR PARTITIONS
كل block يتحقق منه قبل القراءة عبر Merkle tree. تعديل أي byte = رفض القراءة.
↓ يشغّل
🟢 ANDROID FRAMEWORK
مضمون 100% أنه لم يتغير منذ توقيعه.

dm-verity — الحارس البصمي

dm-verity هو Linux kernel module بيستخدم Merkle tree للتحقق من كل block في الـ partition عند القراءة. أي تعديل في أي مكان = error.

dm-verity Merkle Tree — كيف يعمل
            ROOT HASH (مُخزَّن في vbmeta.img)
                       │
              ┌────────┴────────┐
            H(L1a)           H(L1b)
           ┌──┴──┐           ┌──┴──┐
        H(L2a) H(L2b)     H(L2c) H(L2d)
          │      │           │      │
       [B0-B3][B4-B7]   [B8-B11][B12-B15]   ← data blocks (4KB each)

  إذا تغيّر أي block:
  Block 5 تغيّر → H(L2b) مختلف → H(L1a) مختلف → ROOT HASH مختلف
  → dm-verity يرفض القراءة → I/O error → Android يُعلم المستخدم 🚫

🚨
توقيع OTA ≠ توقيع AVB

في مفتاحين مختلفين: مفتاح OTA لتوقيع الـ payload.bin (يتحقق منه update_engine)، ومفتاح AVB لتوقيع الـ partitions نفسها (يتحقق منه bootloader). الاثنين مطلوبين.

OTA Server بـ Node.js/Express — Implementation

ota-server/src/update-controller.ts
import { Request, Response } from 'express'
import { UpdateService } from './update-service'
import { DeviceService } from './device-service'

export class UpdateController {
  constructor(
    private updateService: UpdateService,
    private deviceService: DeviceService
  ) {}

  checkUpdate = async (req: Request, res: Response) => {
    const {
      device_id, device_model, current_build,
      channel, battery_level, is_charging, wifi_connected
    } = req.body

    // 1. سجّل الـ check-in
    await this.deviceService.recordCheckIn(device_id, {
      build: current_build,
      timestamp: new Date()
    })

    // 2. ابحث عن تحديث متاح للـ model والـ channel
    const latestUpdate = await this.updateService.getLatestUpdate({
      model: device_model,
      currentBuild: current_build,
      channel
    })

    if (!latestUpdate) {
      return res.json({ status: 'NO_UPDATE' })
    }

    // 3. تحقق من الـ rollout percentage
    const isEligible = await this.updateService.isDeviceEligible(
      device_id,
      latestUpdate.rollout_percentage
    )

    if (!isEligible) {
      return res.json({ status: 'NO_UPDATE' })
    }

    // 4. فحص شروط التنزيل (بطارية + WiFi)
    if (!latestUpdate.mandatory) {
      if (battery_level < 30 && !is_charging) {
        return res.json({
          status: 'DEFERRED',
          reason: 'LOW_BATTERY',
          retry_after_minutes: 60
        })
      }
    }

    // 5. ابعت التحديث
    const signedUrl = await this.updateService.generateSignedUrl(
      latestUpdate.payload_path,
      3600 // صالح ساعة واحدة
    )

    res.json({
      status: 'UPDATE_AVAILABLE',
      update: {
        ...latestUpdate,
        url: signedUrl
      }
    })
  }
}

Staged Rollout — نشر تدريجي آمن

staged-rollout-strategy.ts
/**
 * Staged Rollout Strategy
 * 1% → 5% → 10% → 25% → 50% → 100%
 * كل مرحلة تستغرق 24-48 ساعة
 */

const ROLLOUT_STAGES = [
  { percentage: 1,   duration_hours: 24,  cohort: 'internal'   },
  { percentage: 5,   duration_hours: 24,  cohort: 'beta'       },
  { percentage: 10,  duration_hours: 48,  cohort: 'early_access'},
  { percentage: 25,  duration_hours: 48,  cohort: 'general'    },
  { percentage: 50,  duration_hours: 24,  cohort: 'general'    },
  { percentage: 100, duration_hours: 0,   cohort: 'all'        }
]

function isDeviceEligible(
  deviceId: string,
  rolloutPercentage: number
): boolean {
  // Hash deterministic: نفس الجهاز دايماً نفس النتيجة
  const hash = murmurhash3(deviceId) % 100
  return hash < rolloutPercentage
}

// مثال: جهاز بـ hash=7 يحصل على التحديث من أول 10%
// جهاز بـ hash=55 ينتظر حتى الـ 50% stage

function selectBootSlot() {
  for (slot of [current, other]) {
    if (slot.bootable && slot.tries > 0) {
      slot.tries--
      return slot
    }
  }
  return recoveryMode()
}

MarkSuccessful — أهم استدعاء في الـ Boot

UpdateVerifier.java — تأكيد نجاح التحديث
// UpdateVerifier يشتغل في الـ background بعد الريبوت
// يتأكد من: DM-verity، system integrity، essential services

public class UpdateVerifier extends IntentService {

  private void verifyUpdateAndMarkSuccessful() {
    // 1. تحقق من dm-verity status
    if (!isDmVerityClean()) {
      Log.e(TAG, "dm-verity corruption detected!")
      triggerRollback()
      return
    }

    // 2. تحقق من الـ essential system processes
    if (!areEssentialServicesRunning()) {
      triggerRollback()
      return
    }

    // 3. ✅ كل شيء تمام — سجّل النجاح
    UpdateEngine updateEngine = new UpdateEngine()
    // هذا يستدعي bootctl mark-boot-successful تحت الغطاء
    updateEngine.resetStatus()
    
    Log.i(TAG, "Update verified successfully! Slot marked as successful.")
  }
}

Streaming OTA — تطبيق بدون تنزيل كامل

ميزة رائعة في Pixel 3+ وAndroid 9+: الـ update_engine يبدأ تطبيق الـ blocks بينما لا يزال التنزيل جارياً. يحتاج HTTP Range Requests من الـ server.

Streaming OTA — الفرق في applyPayload
// Non-streaming: تنزيل كامل أولاً ثم التطبيق
updateEngine.applyPayload(
  "file:///data/ota_update/payload.bin", // local file
  0L, 0L,
  properties
)

// Streaming: التطبيق من URL مباشرة
// update_engine يطلب ranges بـ HTTP Range header
updateEngine.applyPayload(
  "https://cdn.ota.example.com/payload.bin", // remote URL
  payloadOffset,  // offset داخل الـ ZIP
  payloadSize,    // حجم الـ payload.bin
  arrayOf(
    "FILE_HASH=$fileHash",
    "FILE_SIZE=$fileSize",
    "METADATA_HASH=$metaHash",
    "METADATA_SIZE=$metaSize"
  )
)

// الـ server يجب أن يدعم:
// GET /payload.bin HTTP/1.1
// Range: bytes=0-4095          ← block الأول
// Range: bytes=4096-8191       ← block الثاني
// ... يطلب blocks بالترتيب اللي يحتاجه

OTA Best Practices — الملخص النهائي

✅
افعل دايماً
  • دايماً وقّع الـ payload بـ RSA-4096
  • استخدم staged rollout: 1% → 100%
  • راقب crash rates بعد كل مرحلة
  • احفظ target_files لكل build لـ incremental updates
  • استخدم HTTPS فقط لـ payload URLs
  • ضيف expiry للـ signed URLs (max 1 ساعة)
🚫
تجنب دايماً
  • ما تزيدش rollback index إلا بعد التأكد 100%
  • ما تنشرش 100% دفعة واحدة بدون soak testing
  • ما تتجاهلش dm-verity errors في logs
  • ما تعتمدش على userdata في الـ OTA scripts
  • ما تنسىش تحديث care_map للـ incremental updates
🏆
الـ Golden Rule لـ OTA

اللي يميّز فريق OTA المحترف: التحديث الفاشل يجب أن يكون أكثر أماناً من التحديث الناجح. راجع دايماً: هل الجهاز يرجع بأمان للإصدار القديم لو فشل كل شيء؟ لو الجواب لا، الـ system مش ready للـ production.
