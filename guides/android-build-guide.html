01
نظرة تاريخية — Make vs Soong
كيف تطور نظام البناء في Android

نظام Android بدأ يستخدم GNU Make من أول يوم — وكان ذلك كافياً لسنوات. لكن مع نمو AOSP لأكثر من 30 مليون سطر كود، بدأت المشاكل تظهر: البناء بطيء، الأخطاء صعبة التتبع، وإضافة قواعد جديدة معقدة.

🏛️ GNU Make (2008–2021)

نظام بناء قديم، يعتمد على ملفات Android.mk. قوي لكنه بطيء على المشاريع الضخمة. غير متوافق مع البناء المتوازي بكفاءة.

⚡ Soong (2017–الآن)

نظام بناء جديد كتبته Google بلغة Go. يعتمد على Android.bp. أسرع بكثير، ويدعم البناء المتوازي الحقيقي عبر Ninja.

03
Makefile في Android (Legacy)
Android.mk — النظام القديم الذي لا تزال بعض الأجهزة تستخدمه

رغم أن Make أصبح Legacy، فهم Android.mk ضروري لأن كثيراً من الأجهزة القديمة والـ Custom ROM لا تزال تعتمده.

Android.mk — بناء مكتبة C
# تعريف نقطة البداية لكل وحدة جديدة

LOCAL_PATH := $(call my-dir)

include $(CLEAR_VARS)



# اسم الوحدة (Module)

LOCAL_MODULE              := libmymodule



# الكود المصدري

LOCAL_SRC_FILES           := \

    src/main.cpp          \

    src/helper.cpp



# مكتبات يعتمد عليها

LOCAL_SHARED_LIBRARIES    := liblog libcutils

LOCAL_STATIC_LIBRARIES    := libbase



# Flags للمترجم

LOCAL_CFLAGS              := -Wall -Werror -O2

LOCAL_CPPFLAGS            := -std=c++17



# Header files

LOCAL_C_INCLUDES          := $(LOCAL_PATH)/include



# أين يُثبَّت في الجهاز

LOCAL_MODULE_PATH         := $(TARGET_OUT_SHARED_LIBRARIES)



# نوع الـ Module (shared library)

include $(BUILD_SHARED_LIBRARY)
Android.mk — بناء APK مدمج
include $(CLEAR_VARS)



LOCAL_MODULE              := MySystemApp

LOCAL_MODULE_CLASS        := APPS

LOCAL_MODULE_SUFFIX       := .apk



# APK مبني مسبقاً (Prebuilt)

LOCAL_SRC_FILES           := MyApp.apk

LOCAL_CERTIFICATE         := platform  # توقيع النظام

LOCAL_PRIVILEGED_MODULE   := true



# مكان التثبيت: /system/priv-app/

include $(BUILD_PREBUILT)
الكلمة المفتاحيةالمعنى
BUILD_SHARED_LIBRARYبناء مكتبة .so مشتركة
BUILD_STATIC_LIBRARYبناء مكتبة .a ثابتة
BUILD_EXECUTABLEبناء ملف تنفيذي binary
BUILD_PACKAGEبناء APK من الكود
BUILD_PREBUILTتثبيت ملف مبني مسبقاً
CLEAR_VARSمسح متغيرات الوحدة السابقة

Blueprint & Android.bp
بنية الملف بالتفصيل
Android.bp — المفاهيم الأساسية
// 1. تعريف متغيرات (Variables)

my_sources = ["a.cpp", "b.cpp"]



// 2. تعريف Defaults (قواعد مشتركة)

cc_defaults {

    name: "my_defaults",

    cflags: ["-DDEBUG"],

    shared_libs: ["liblog"],

}



// 3. استخدام الـ Defaults

cc_binary {

    name: "myapp",

    defaults: ["my_defaults"],  // يرث القواعد

    srcs: my_sources,

}



// 4. Conditions بناءً على النظام

cc_library {

    name: "libplatform",

    

    target: {

        android: {

            srcs: ["android/platform.cpp"],

            shared_libs: ["libbinder"],

        },

        linux: {

            srcs: ["linux/platform.cpp"],

        },

        host: {

            srcs: ["host/platform.cpp"],

        },

    },

    

    // بناءً على Architecture

    arch: {

        arm: { cflags: ["-marm"] },

        arm64: { cflags: ["-march=armv8-a"] },

        x86_64: { cflags: ["-m64"] },

    },

}
✅
الفرق الجوهري: Soong محدد النوع

في Soong كل شيء محدد النوع — لا يمكن وضع قيمة خاطئة. مثلاً vendor_available يقبل فقط true/false، وليس نصاً. هذا يلغي كثيراً من الأخطاء الشائعة في Make.

06
Ninja — المحرك الخفي
النظام الذي ينفذ البناء الفعلي

Ninja ليس للكتابة اليدوية — هو ينفذ فقط. Soong يولّد ملف build.ninja ثم Ninja ينفذه. فكره في الأساس: بناء متوازي 100% مع تتبع دقيق للتبعيات.

build.ninja — مثال (مولّد تلقائياً)
# هذا يُولَّد تلقائياً — لا تكتبه يدوياً



rule cc

  command = $cc $cflags -c $in -o $out

  description = CC $out



rule link

  command = $linker $in -o $out $ldflags



build main.o: cc src/main.cpp

  cflags = -Wall -O2



build helper.o: cc src/helper.cpp

  cflags = -Wall -O2



build myapp: link main.o helper.o
⚡
لماذا Ninja أسرع من Make؟

Ninja يُحلّل كل التبعيات مرة واحدة مسبقاً، ثم يُنفّذ كل شيء ممكن بالتوازي. Make يُحلّل بشكل متسلسل. في AOSP: Make يأخذ 3+ ساعات، Ninja يأخذ 45 دقيقة على نفس الجهاز.

07
Kati — الجسر بين المنظومتين
التوافقية مع الكود القديم

Kati هو محاكي GNU Make مكتوب بـ C++ وGo. مهمته: تحويل ملفات Android.mk القديمة إلى ملفات Ninja بدلاً من تنفيذها مباشرة. هذا يجعل البناء أسرع حتى للكود القديم.

Flashing & Bootloader
تثبيت البيلد على الجهاز — الطريقة الصحيحة
🔒
Bootloader Lock/Unlock — الأساس

الـ Bootloader هو أول ما يعمل عند تشغيل الهاتف. "مقفل" = لا يقبل إلا Android الرسمي. "مفتوح" = يقبل أي بيلد. فتح الـ Bootloader يمسح كل البيانات ويُبطل الضمان في أغلب الأجهزة.

1

تفعيل Developer Options وفتح Bootloader

Settings → About Phone → اضغط Build Number 7 مرات → Developer Options → Enable OEM Unlocking

2

الدخول لـ Fastboot Mode

أوقف الهاتف + اضغط Power + Volume Down (يختلف حسب الجهاز)

3

فتح الـ Bootloader

fastboot flashing unlock — ستظهر رسالة تأكيد على الشاشة

4

Flash البيلد الجديد

استخدم أوامر fastboot لتثبيت الصور الجديدة

bash — Fastboot Commands الأساسية
# التحقق من الاتصال

fastboot devices



# معلومات عن الجهاز

fastboot getvar all



# ============ Flash Partitions فردية ============



# Boot partition (Kernel)

fastboot flash boot boot.img



# Recovery

fastboot flash recovery recovery.img



# System

fastboot flash system system.img



# Vendor

fastboot flash vendor vendor.img



# Verified Boot Metadata

fastboot flash vbmeta vbmeta.img



# ============ Flash كامل (الأسهل) ============



# يثبت كل الصور دفعة واحدة

fastboot flashall



# مع مسح البيانات (-w = wipe)

fastboot flashall -w



# ============ Dynamic Partitions (Android 10+) ============



# Super partition (يحتوي system + vendor + product)

fastboot flash super super.img



# أو لإعادة التحجيم الديناميكي

fastboot wipe-super super_empty.img

fastboot flash system system.img

fastboot flash vendor vendor.img



# ============ إعادة التشغيل ============

fastboot reboot                  # تشغيل عادي

fastboot reboot-bootloader       # العودة لـ Fastboot

fastboot reboot recovery         # الدخول للـ Recovery

# في حال استخدام Custom Recovery مثل TWRP

💡
fastboot vs adb sideload

fastboot: يعمل في Bootloader Mode، يثبت صور قسم كاملة (.img). أسرع وأكثر تحكماً. | adb sideload: يعمل في Recovery Mode، يثبت ملفات ZIP. أسهل للمستخدم العادي.

الـ Module المطلوب غير موجود أو اسمه خاطئ. تحقق من اسم الـ module في Android.bp أو Android.mk.

❌ خاطئ
mm libMyLib

// Module اسمه في Android.bp: "libmylib"
✅ صحيح
mm libmylib

// الاسم بالضبط كما في Android.bp
❌ Out of memory / Java heap space

ذاكرة غير كافية. الحل: export _JAVA_OPTIONS="-Xmx8g" لزيادة heap Java. أو أضف swap: sudo fallocate -l 16G /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile

❌ fastboot: error: FAILED (remote: 'Partition not found')

اسم الـ partition خاطئ أو الجهاز يستخدم Dynamic Partitions. جرب fastboot flash super super.img بدلاً من flash system/vendor منفصلة.

❌ Bootloop بعد الـ Flash

السبب الأكثر شيوعاً: عدم توافق الـ vendor blobs مع ROM الجديد. الحل: استخرج الـ blobs من نفس إصدار Android المستهدف، أو استخدم الـ factory image المطابق.

❌ selinux: avc denied

خطأ صلاحيات SELinux. إضافة policy في sepolicy/ أو مؤقتاً: adb shell setenforce 0 (للاختبار فقط، ليس للإنتاج).
