ما هو SELinux؟

نظام أمان متقدم مدمج في نواة Linux يطبق مبدأ أقل الصلاحيات

🛡️

SELinux = Security-Enhanced Linux

طوّرته NSA (وكالة الأمن القومي الأمريكية) وتبرّعت به لمجتمع Linux عام 2000. الآن جزء أساسي من Red Hat / Fedora / CentOS / AlmaLinux / Rocky Linux.

🔒

الفلسفة الأساسية

بدلاً من "كل شيء مسموح ما لم يُمنع" — SELinux يقول "كل شيء ممنوع ما لم يُسمح". حتى لو اخترق المهاجم التطبيق، لن يستطيع تجاوز السياسة.

⚡

MAC vs DAC

DAC (التقليدي): الملفات تحكمها أذونات owner/group/other.
MAC (SELinux): النواة نفسها تحكم كل وصول بناءً على labels وسياسات مكتوبة مسبقاً.

🏗️

أين يعمل؟

يعمل داخل نواة Linux على مستوى LSM (Linux Security Module). كل استدعاء نظام (syscall) يمرّ عبر SELinux قبل التنفيذ.

لماذا يهرب الناس من SELinux؟

المشكلة الأكثر شيوعاً: عند ظهور خطأ "Permission denied"، يكتب 90% من المبتدئين setenforce 0 أو SELINUX=disabled! هذا خطأ فادح. الحل الصحيح هو فهم الرسالة وإصلاح السياسة.

🔍

ترتيب الفحص مهم!

SELinux يفحص بعد DAC. يعني: إذا رفض DAC الوصول، SELinux لن يُشغَّل أصلاً. لكن إذا سمح DAC ورفض SELinux — الرفض النهائي. كلاهما يجب أن يسمح.

⚡ Access Vector Cache (AVC) — قلب SELinux

1. Cache Miss

أول طلب لـ context+permission: يُرسَل لمحرك السياسة، يُحسَب القرار، يُخزَّن في AVC

2. Cache Hit

الطلبات التالية نفسها: القرار يُسترجع من AVC مباشرة — سريع جداً (microseconds)

3. Cache Invalidation

عند تغيير السياسة أو الـ context، الـ AVC يُمسح ويُعاد الحساب

🧬 وراثة السياق (Context Inheritance)

كيف يرث الملف سياقه؟
# المجلد الأب يحدد سياق الملفات الجديدة
$ ls -dZ /var/www/html
system_u:object_r:httpd_sys_content_t:s0 /var/www/html/

# ملف جديد يرث سياق مجلده
$ touch /var/www/html/newfile.html
$ ls -Z /var/www/html/newfile.html
unconfined_u:object_r:httpd_sys_content_t:s0 newfile.html
✓ ورث httpd_sys_content_t من المجلد الأب

# لكن ملف منقول (moved/copied) يحتفظ بسياقه!
$ cp /tmp/file.txt /var/www/html/
$ ls -Z /var/www/html/file.txt
unconfined_u:object_r:user_tmp_t:s0 file.txt
✗ احتفظ بسياق /tmp (خطأ شائع!)

# الحل: restorecon لإصلاح السياق
$ restorecon /var/www/html/file.txt
$ ls -Z /var/www/html/file.txt
unconfined_u:object_r:httpd_sys_content_t:s0 file.txt
✓ تم إصلاح السياق
الأوضاع

أوضاع SELinux الثلاثة

اختر الوضع المناسب لكل بيئة

🔒
Enforcing
SELinux مفعّل بالكامل. يمنع كل انتهاك ويسجله. للإنتاج.
👁️
Permissive
يسجل الانتهاكات فقط بدون منعها. للتطوير والاختبار.
⛔
Disabled
SELinux معطّل تماماً. خطير — تجنّبه في الإنتاج.
إدارة الأوضاع
# فحص الوضع الحالي
$ getenforce
Enforcing

$ sestatus
SELinux status:                 enabled
SELinuxfs mount:                /sys/fs/selinux
SELinux mount point:            /sys/fs/selinux
Loaded policy name:             targeted
Current mode:                   enforcing
Mode from config file:          enforcing
Policy MLS status:              enabled
Policy deny_unknown status:     allowed
Memory protection checking:     actual (secure)
Max kernel policy version:      33

# تغيير مؤقت (حتى إعادة التشغيل)
$ setenforce 0   # Permissive
$ setenforce 1   # Enforcing

# تغيير دائم (يحتاج reboot)
$ vi /etc/selinux/config

SELINUX=enforcing   # 🔒 للإنتاج
SELINUX=permissive  # 👁️ للاختبار
SELINUX=disabled    # ⚠️ تجنّب!

SELINUXTYPE=targeted  # نوع السياسة (targeted/mls)

# وضع permissive لعملية واحدة فقط (بدون إيقاف الكل!)
$ semanage permissive -a httpd_t
$ semanage permissive -d httpd_t   # للحذف
$ semanage permissive -l           # للعرض
🚨

تحذير: الانتقال من Disabled إلى Enforcing

إذا كان SELinux معطّلاً وأعدت تفعيله، يجب إعادة تسمية (relabel) كل الملفات أولاً! أنشئ الملف /.autorelabel ثم أعد التشغيل. بدون هذه الخطوة النظام قد لا يعمل!

إعادة التسمية بعد تفعيل SELinux
# تغيير من disabled إلى enforcing بأمان

# 1. عدّل /etc/selinux/config
$ sed -i 's/SELINUX=disabled/SELINUX=enforcing/' /etc/selinux/config

# 2. أنشئ ملف autorelabel
$ touch /.autorelabel

# 3. أعد التشغيل (سيأخذ وقتاً أطول للـ relabeling)
$ reboot

# أو يدوياً بدون reboot (أبطأ)
$ restorecon -Rv /
$ setenforce 1
السياسات

سياسات SELinux وكتابتها

من المدمجة إلى المخصصة — كل مستويات السياسات

🎯

Targeted Policy

الافتراضي في RHEL/Fedora. يُقيّد فقط الخدمات الحساسة (httpd, sshd, mysqld...) وتترك باقي العمليات unconfined.

الأكثر استخداماً
🔐

MLS Policy

Multi-Level Security. يُقيّد كل شيء بمستويات أمان (s0-s15). للحكومات والعسكريين والبيئات عالية الأمان.

حكومي/عسكري
🧩

Minimum Policy

نسخة مخففة من Targeted. فقط القواعد الأساسية. للأنظمة المدمجة (Embedded) والبيئات المحدودة.

Embedded

✍️ كتابة سياسة مخصصة — من الصفر

🟢 مستوى مبتدئ

الطريقة الآلية: audit2allow

أسرع طريقة: خذ رسائل الرفض من log وحوّلها تلقائياً لسياسة

audit2allow — الطريقة السريعة
# 1. شغّل تطبيقك وخلّيه يفشل
$ systemctl start myapp
Job failed. See systemctl status myapp

# 2. اعرض رسائل الرفض
$ ausearch -m avc -ts recent
type=AVC msg=audit(1234567890.123:456): avc: denied { read } for
pid=1234 comm="myapp" name="config.cfg" dev="sda1" ino=789
scontext=system_u:system_r:myapp_t:s0
tcontext=system_u:object_r:etc_t:s0 tclass=file permissive=0

# 3. أنشئ السياسة تلقائياً
$ ausearch -m avc -ts recent | audit2allow -M mypolicy

# يُنشئ ملفين:
mypolicy.te  ← ملف السياسة النصي
mypolicy.pp  ← السياسة المُصرَّفة

# 4. ثبّت السياسة
$ semodule -i mypolicy.pp
تم تثبيت السياسة بنجاح

# تحقق من التثبيت
$ semodule -l | grep mypolicy
mypolicy    1.0
🟡 مستوى متوسط

كتابة ملف .te يدوياً

myapp.te — ملف Type Enforcement
## MyApp SELinux Policy Module
## Allows myapp to read config files in /etc

# تعريف النوع الجديد وربطه بـ domain
policy_module(myapp, 1.0)

type myapp_t;
type myapp_exec_t;
domain_type(myapp_t)
domain_entry_file(myapp_t, myapp_exec_t)

# السماح لـ myapp بقراءة /etc
allow myapp_t etc_t:file { read open getattr };

# السماح بكتابة logs
allow myapp_t var_log_t:file { write create append open };

# السماح بالاستماع على منفذ TCP
allow myapp_t myapp_port_t:tcp_socket { listen accept };

# السماح باستخدام الشبكة
corenet_tcp_sendrecv_generic_if(myapp_t)
corenet_tcp_bind_generic_node(myapp_t)
تصريف وتثبيت السياسة
# تجميع ملف .te
$ make -f /usr/share/selinux/devel/Makefile myapp.pp
Compiling targeted myapp module
Creating targeted myapp.pp policy package
rm tmp/myapp.mod.fc tmp/myapp.mod

# تثبيت الوحدة
$ semodule -i myapp.pp

# التحقق
$ semodule -l | grep myapp
myapp   1.0

# حذف الوحدة (عند الحاجة)
$ semodule -r myapp
🟠 مستوى متقدم

ملف .fc لتسمية الملفات + ملف .if للواجهات

myapp.fc — File Contexts
# يُحدّد أي ملفات تأخذ أي سياق

/usr/bin/myapp         -- system_u:object_r:myapp_exec_t:s0
/etc/myapp(/.*)?       -- system_u:object_r:myapp_conf_t:s0
/var/lib/myapp(/.*)?   -- system_u:object_r:myapp_var_t:s0
/var/log/myapp\.log    -- system_u:object_r:myapp_log_t:s0
/run/myapp\.pid        -- system_u:object_r:myapp_var_run_t:s0
myapp.if — Interfaces (للوحدات الأخرى)
## السماح للـ domain الآخر بقراءة ملفات myapp
interface(`myapp_read_config',`
    gen_require(`
        type myapp_conf_t;
    ')
    read_files_pattern($1, myapp_conf_t, myapp_conf_t)
')

## السماح لـ admin بإدارة myapp
interface(`myapp_admin',`
    gen_require(`
        type myapp_t, myapp_conf_t;
    ')
    allow $1 myapp_t:process signal_perms;
    myapp_read_config($1)
')
البوليانز

SELinux Booleans — مفاتيح التحكم

بدون كتابة سياسة — غيّر سلوك SELinux بمفتاح واحد

ما هي البوليانز؟

هي متغيرات on/off مدمجة في السياسة تتحكم في سلوكها. بدلاً من تعديل ملفات .te، تقلب switch واحد.

إدارة البوليانز
# عرض كل البوليانز
$ getsebool -a
httpd_can_network_connect      --> off
httpd_can_sendmail              --> off
httpd_enable_homedirs           --> off
httpd_use_nfs                   --> off

# البحث عن بوليان
$ getsebool -a | grep httpd
$ semanage boolean -l | grep httpd

# تفعيل مؤقت
$ setsebool httpd_can_network_connect 1

# تفعيل دائم (P = Persistent)
$ setsebool -P httpd_can_network_connect on
$ setsebool -P httpd_can_sendmail on

# تعطيل
$ setsebool -P httpd_can_network_connect off

أهم البوليانز وحالات استخدامها

البوليان الافتراضي متى تفعّله؟
httpd_can_network_connectoffعندما Apache يحتاج يتصل بخدمة خارجية (proxy, API)
httpd_can_network_connect_dboffعندما PHP/Apache يتصل بـ MySQL/PostgreSQL بعيد
httpd_enable_homedirsoffلقراءة مجلدات المستخدمين في /home/user/public_html
httpd_use_nfsoffعندما محتوى الويب على NFS mount
httpd_can_sendmailoffللتطبيقات التي ترسل email عبر Apache
httpd_execmemoffبعض مفسّرات PHP تحتاجه (php-fpm مع OPcache)
samba_enable_home_dirsoffمشاركة مجلدات home عبر Samba
allow_user_exec_contentonالسماح للمستخدمين بتشغيل سكريبتات في home
ftp_home_diroffFTP يصل لمجلدات home
ssh_sysadm_loginoffتسجيل SSH مباشر لـ sysadm role

🔧 خطوات التشخيص

الخطوات المنهجية لحل مشكلة SELinux
━━━ الخطوة 1: تأكد أن SELinux هو السبب ━━━
$ setenforce 0
$ systemctl restart myservice
# إذا نجح الآن → SELinux هو السبب
$ setenforce 1   # أعد Enforcing فوراً!

━━━ الخطوة 2: اقرأ رسائل الرفض ━━━
$ ausearch -m avc -ts recent
$ journalctl | grep SELinux
$ sealert -a /var/log/audit/audit.log
$ cat /var/log/audit/audit.log | grep avc

━━━ الخطوة 3: حدد نوع المشكلة ━━━

# مشكلة context في ملف؟
$ ls -Z /path/to/file
# إذا الـ type غلط، أصلحه:
$ chcon -t httpd_sys_content_t /var/www/html/myfile
$ restorecon /var/www/html/myfile

# مشكلة context في مجلد؟
$ semanage fcontext -a -t httpd_sys_content_t "/mywebroot(/.*)?"
$ restorecon -Rv /mywebroot

# مشكلة منفذ؟
$ semanage port -l | grep 8080
$ semanage port -a -t http_port_t -p tcp 8080

# مشكلة boolean؟
$ getsebool -a | grep httpd
$ setsebool -P httpd_can_network_connect on

━━━ الخطوة 4: اصنع سياسة مخصصة إذا لزم ━━━
$ ausearch -m avc -ts recent | audit2allow -M myfix
$ semodule -i myfix.pp

🚑 مشاكل شائعة وحلولها

⛔ Apache لا يقرأ ملفات من مجلد غير /var/www

# المشكلة: نقلت site إلى /data/mysite
AVC: denied { read } httpd_t /data/mysite

# الحل
$ semanage fcontext -a -t \
    httpd_sys_content_t \
    "/data/mysite(/.*)?"
$ restorecon -Rv /data/mysite

⛔ Apache لا يتصل بـ Database

# المشكلة: PHP يتصل بـ MySQL
AVC: denied { name_connect } httpd_t

# الحل
$ setsebool -P \
    httpd_can_network_connect_db on
# أو للاتصال العام:
$ setsebool -P \
    httpd_can_network_connect on

⛔ Nginx على منفذ 8080

# المشكلة: bind failed
AVC: denied { name_bind } httpd_t port 8080

# تحقق أولاً
$ semanage port -l | grep 8080

# الحل: أضف المنفذ
$ semanage port -a -t \

⛔ Script لا يعمل في cron

# أو للملفات في /etc/cron.d
$ restorecon /etc/cron.d/myjob

🟠 متقدم

MCS — Multi-Category Security

يستخدم categories (c0..c1023) لعزل حاويات/VM بعضها عن بعض. هذا ما تستخدمه Docker وKubernetes داخلياً!

# Context بـ MCS categories
system_u:system_r:svirt_t:s0:c1,c100
system_u:system_r:svirt_t:s0:c2,c200
# لا يمكن لـ c1,c100 أن يصل لـ c2,c200
# حتى لو نفس الـ type!

# قاعدة المجال
s0:c1,c100 يقرأ s0:c1,c100 ✓
s0:c1,c100 لا يقرأ s0:c2,c200 ✗
🔴 خبير

MLS — Multi-Level Security

Bell-LaPadula model. مستويات s0→s15. "اقرأ للأسفل، اكتب للأعلى". للحكومات والعسكريين.

# مستويات MLS
s0 = Unclassified (عام)
s1 = Confidential (سري)
s2 = Secret (سري للغاية)
s3 = Top Secret (محرم جداً)

# قاعدة Bell-LaPadula
s2 يقرأ s0, s1, s2 ✓
s2 لا يقرأ s3 ✗ (no read-up)
s2 يكتب s2, s3 ✓
s2 لا يكتب s1, s0 ✗ (no write-down)

🐳 SELinux + Containers (Docker/Podman)

SELinux في بيئة الحاويات
# Docker يستخدم svirt_sandbox_file_t للـ volumes
$ docker run -v /mydata:/data:Z myimage
# Z = relabel للـ container وحده (ثلاثي الـ category)
# z = relabel مشترك بين كل الـ containers

# Podman مع SELinux
$ podman run --security-opt label=type:svirt_t \
    -v /mydata:/data:Z myimage

# Kubernetes SELinux context
apiVersion: v1
kind: Pod
spec:
  securityContext:
    seLinuxOptions:
      level: "s0:c123,c456"
  containers:
  - name: myapp
    securityContext:
      seLinuxOptions:
        type: "httpd_t"

# تعطيل SELinux لـ container معين (ليس للكل!)
$ podman run --security-opt label=disable myimage

اختبار سريع

❓ موقع Apache يعطي خطأ "403 Forbidden" بعد نقل الملفات من /tmp إلى /var/www/html. ما السبب المرجّح؟
أ
مشكلة في أذونات chmod
ب
الملفات احتفظت بسياق /tmp (user_tmp_t) بدل httpd_sys_content_t
ج
Apache غير مشغّل
د
مشكلة في /etc/hosts
✅ إجابة صحيحة! عند نقل/نسخ ملف، يحتفظ بسياقه الأصلي. الحل: restorecon -Rv /var/www/html
❓ تريد أن تسمح لـ Apache بالاتصال بقاعدة بيانات MySQL على سيرفر آخر. أفضل طريقة؟
أ
setenforce 0
ب
كتابة سياسة مخصصة كاملة
ج
setsebool -P httpd_can_network_connect_db on
د
إضافة منفذ 3306 بـ semanage port
✅ إجابة صحيحة! هذه حالة مناسبة تماماً للـ Boolean. سريع، آمن، ومصمّم خصيصاً لهذا الغرض.
❓ ما الفرق بين chcon و semanage fcontext + restorecon؟
أ
لا فرق، كلاهما نفس النتيجة
ب
chcon مؤقت (يُعاد بعد relabel)، semanage دائم ومُسجَّل في قاعدة البيانات
ج
chcon للمجلدات، semanage للملفات
د
semanage أسرع من chcon
✅ إجابة صحيحة! chcon يغيّر الـ label مباشرة لكن بعد relabel يعود للقديم. semanage يُضيف قاعدة دائمة، ثم restorecon يطبّقها.
